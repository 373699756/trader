# 实时荐股与 DeepSeek 高效协同优化计划

状态：非权威执行计划，尚未实施。

产品、架构、运行、API、运维和验收以
[软件业务设计文档](software-business-design.md) 为唯一权威；候选、过滤、因子、评分、
风险、DeepSeek、融合、动作和排名以
[荐股策略文档](recommendation-strategy.md) 为唯一权威。本文只记录后续独立交付批次，
不得覆盖两份权威文档；实施引起契约变化时必须先更新对应权威文档。

## 1. 目标与边界

- 本地规则继续负责候选、硬过滤、基础评分、本地风险、动作和 TopK；DeepSeek 只处理
  本地难以可靠判断的新闻催化、预期差、证据冲突和潜在风险。
- `long` 永久保持纯本地，只展示当前观察结果，不调用 DeepSeek、不冻结、不产生动作、
  不写推荐历史。
- today、tomorrow 和 d25 共用同一股票的原始 DeepSeek 事实，再按各策略权重在本地投影；
  禁止为相同证据重复请求。
- 每日 188 次物理 HTTP 请求仍是不可突破的全局硬上限；正常日按高价值候选控制在最多
  58 次主审、包含 Pro 时最多 66 次、包含 emergency 时计划内最多 71 次。
- 本计划不建设历史 60 日回补、300 条配对、Bootstrap、shadow、自动调权或在线学习。
  这些工作作为后续独立批次；没有点时样本外证据前，不宣称提高实际荐股收益。

## 2. 本地主链与固定融合

保持三板候选、硬过滤、流动性、趋势、同行、量价、稳定性、下行保护、动作门槛和集中度
约束。DeepSeek 不得新增候选、解除硬过滤、直接给出生产动作、目标价或最终排名，也不得
在失败时阻塞本地推荐。

融合公式保持：

```text
final_score = clamp(
    local_score * 0.68
    + deepseek_score * 0.32
    - deepseek_risk_penalty,
    0,
    100,
)
```

`local_score` 已扣本地风险，融合时不得再次扣除。最终继续使用 `ROUND_HALF_UP` 保留两位
小数，固定验收向量结果保持 `83.40`。DeepSeek 不可用、超时、预算耗尽或证据不足时，
整条链路回退本地推荐并显式标记降级。

## 3. 新闻、公告与结构化事实

### 3.1 新闻利好与负面事实

本地先完成点时过滤、来源分级和事件去重，DeepSeek 再判断：

- 事件为正向、负向或中性，重要度为高、中或低；
- 事件已经确认、仍待确认或仅为传闻；
- 价格尚未反映、部分反映或已经充分反映；
- 影响周期更适合 today、tomorrow 还是 d25；
- 是否存在相反证据、事实冲突或潜在风险。

只有官方来源或至少两个独立可信来源支持的正向事实才允许加分。单一软新闻、转载、传闻、
标题情绪和模型自报置信度不得加分。同一事件只能进入一个正向评分通道，避免本地新闻情绪
与 DeepSeek 催化重复加分。官方负面事实通过结构化 schema、证据引用、有效期和本地风险
规则后，可以扣分或 veto。

### 3.2 `deepseek_review_v4`

V4 输出限制为以下结构化事实：

- 催化方向、重要度、确认状态、作用周期和证据引用；
- 价格反映状态；
- 基本面改善、稳定、恶化或未知；
- 行业政策正向、中性、负向或未知；
- 监管、减持、解禁、质押、诉讼和业绩风险；
- 证据冲突、覆盖完整度和 `abstain`。

模型不得输出生产扣分、veto、目标价、动作、排名或最终分。自由文本、评级、情绪形容词和
模型自报置信度只供脱敏审计，不能进入生产计算。

### 3.3 本地确定性映射

各维度从 50 分开始，由本地纯函数映射并限制在 30-70：

| 维度 | 本地映射 |
| --- | --- |
| 催化重要度 | 高/中/低按事实方向记 `±15/±8/±3` |
| 预期差 | 未反映 `±12`、部分反映 `±5`、充分反映 `0` |
| 基本面 | 改善 `+15`、稳定 `+3`、恶化 `-18` |
| 行业政策 | 正向 `+10`、中性 `0`、负向 `-12` |
| 风险质量 | 只表达覆盖质量；实际风险进入独立 penalty/veto |

证据质量系数固定为：官方 `1.00`、两个独立可信来源 `0.85`、单一可信来源 `0.65`、
软信息 `0.40`、存在冲突时最高 `0.25`。维度结果按质量系数向 50 收缩。至少三个维度
有效且加权证据覆盖不低于 `0.60` 时才应用模型分，否则使用本地结果。

| 策略 | 催化 | 预期差 | 基本面 | 行业政策 | 风险质量 |
| --- | ---: | ---: | ---: | ---: | ---: |
| today | 30% | 25% | 10% | 15% | 20% |
| tomorrow | 25% | 25% | 15% | 15% | 20% |
| d25 | 20% | 20% | 25% | 20% | 15% |

实际风险只进入 `deepseek_risk_penalty` 和本地映射的 veto，避免在风险维度与融合扣分中
重复计算。

## 4. 请求时间、用途与停止条件

下表是正常软上限，不是必须耗完的配额；没有合格候选时调用数为 0：

| 用途 | 正常软上限 | 使用时间 | 触发条件 |
| --- | ---: | --- | --- |
| 共享预热 | 10 | 09:15-09:29:30 | 昨日正式/观察 Top18、盘前新公告或新风险；只生成原始事实缓存 |
| today 主审 | 22 | 09:30-11:18 | today 数据完整后的动作门槛、TopK 边界、新催化、新风险或证据冲突 |
| tomorrow 主审 | 14 | 13:00-14:48 | 尾盘特征形成后复核 tomorrow 及其与 d25 共享的候选 |
| d25 主审 | 12 | 13:00-14:48 | 只复核 d25 独有且未被 tomorrow 请求或共享缓存覆盖的候选 |
| Pro 追加复核 | 8 | Flash 完成后至所属策略截止前 | 可能改变动作、重大证据冲突、高风险 veto 或下行保护冲突 |
| emergency | 5 | 所属策略正常窗口内 | 正常软上限耗尽后，仅处理新重大风险或冻结保护集合变化 |

### 4.1 共享预热

- 只在盘前执行，输入为昨日正式推荐、观察池及盘前出现新官方公告的股票。
- 只提取公司、公告、政策和风险事实，不使用尚未形成的盘中量价判断，也不发布评分。
- 缓存命中后，today、tomorrow 和 d25 不重复调用。
- 09:30 后不得补耗未使用的共享预热额度；盘中启动直接进入对应策略主审。

### 4.2 today 主审

第一次调用必须同时满足：当日有效候选已经形成、本地评分完成、行情和证据有效、股票进入
高价值复核集合。后续只有出现以下变化才能再次入队：

- 新的证据 manifest；
- 价格跨越新的反应桶；
- 新风险可能触发 veto；
- 股票新进入动作门槛 `±5` 分或 TopK 边界。

11:18 停止提交，预留两分钟完成返回校验和 11:20 冻结。迟到结果只记脱敏审计，不得修改
冻结记录。

### 4.3 tomorrow 与 d25 主审

- 13:00 后开始，避免上午量价尚未形成时浪费请求。
- tomorrow 和 d25 先联合去重；两者共享的原始事实请求归属 tomorrow，d25 直接复用。
- d25 额度只用于 d25 独有的中期基本面、趋势和政策证据。
- 证据 manifest 和价格反应桶未变化时不得重复请求。
- 14:48 停止提交，14:50 冻结；迟到结果不得修改正式记录。

### 4.4 Pro 追加复核

Flash 结果满足以下任一条件才允许调用 Pro：

- 使用 Flash 后，股票将在可执行与不可执行之间切换；
- 官方正面与官方负面材料冲突；
- 新风险可能触发 veto；
- DeepSeek 方向与本地下行保护冲突；
- Flash 对重大事实给出互相矛盾且无法由本地规则消解的判断。

Pro 只能确认、删除或降低正向事实，不能解除硬过滤、取消本地风险或新增无证据利好。
Pro 请求计入每日 `Pro <= 8`，同时计入来源策略硬桶；全局物理请求只计一次，不和主审
重复计数，也不得绕过 11:18/14:48 截止。

### 4.5 emergency

只有同时满足以下条件才能使用：

- 对应策略正常软上限已经耗尽；
- 出现新的官方重大负面事实，或冻结保护集合因新风险发生变化；
- 缓存中没有同一证据版本的有效结果；
- 仍处于所属策略正常提交截止时间以前。

emergency 不得用于普通利好、提高排名、补满 TopK、低价值重试或消耗剩余额度。每次请求
同时计入 emergency 独立原子桶、来源策略硬桶和 188 次全局物理请求。

## 5. 高价值候选与预算计数

### 5.1 候选优先级

today、tomorrow 和 d25 联合构造待复核集合并按股票去重，只复核：

1. 新官方重大风险；
2. 新官方重大催化；
3. 距动作门槛不超过 5 分的候选；
4. 本地 Top18 或 Top10 边界前后 2 名；
5. 正反证据冲突；
6. 本地量价方向与事件方向冲突；
7. 19-30 名中出现新公告、财务变化或重大政策证据者。

硬过滤失败、低可靠度、数据不完整以及明显远离排名边界的候选不得调用 DeepSeek。

### 5.2 软目标与硬边界

- 每次真实 HTTP 请求计为一次物理请求。
- Flash、Pro、超时、失败、重试和 schema 修复都计数。
- 缓存命中、本地投影和本地回退不计数。
- 单次请求包含 1-8 股时仍只计一次物理请求，Pro 每批最多 4 股。
- 完全不可解析时整批最多修复一次；单股错误只修复错误子集。
- 达到软上限后停止普通调用，未使用额度不跨策略转移。
- emergency 不能突破策略截止时间、策略硬桶或全局硬上限。

计划实施后的硬桶为：

| 硬桶 | 上限 |
| --- | ---: |
| today | 70 |
| tomorrow | 45 |
| d25 | 35 |
| long | 0 |
| shared_preheat | 15 |
| emergency | 5 |
| 全局物理请求 | 188 |

`188` 是全局灾难保护上限，不代表各桶必须合计用满。正常主审软目标合计最多 58 次；
包含 Pro 时最多 66 次；包含 emergency 的计划内最坏情况为 71 次。

## 6. Flash、Pro、批处理与缓存

- V4-Flash 使用非思考模式完成普通事实提取；V4-Pro 只处理预注册的冲突、高风险或动作
  边界案例。
- 每股最多发送 12 条证据：风险 3、财务 2、新闻公告 5、结构化行情 1、反证 1。
- 单股输入目标不超过 1200 token，整批不超过约 12000 token；共享市场背景、schema 和
  规则每批只发送一次。
- 使用 1-8 股自适应批次，常规请求优先凑满至少 2 股；候选只剩 1 股或 emergency 时允许
  单股提交。输出上限按批次规模调整，Pro 每批最多 4 股。
- JSON 根结构有效时逐股接受并缓存合法结果，只修复错误股票；长度截断时保留已经完整结束
  的结果；完全不可解析时整批最多修复一次。
- 原始事实缓存键包含股票、交易日、证据 manifest、价格反应桶、模型、prompt 和 schema，
  不包含策略、排名、动作阈值或普通 merge epoch。
- 策略投影缓存包含策略和策略版本，但不重复调用 DeepSeek。事实、价格反应桶、模型、
  prompt 或 schema 变化时才失效。
- 冻结后迟到结果不得改变正式记录；15:00 收盘补算保持纯本地；Web 只读取已发布结果，
  不触发 DeepSeek、行情抓取或评分。

## 7. 接口与可观测性

- `DeepSeekReviewPort` 保持批量调用形式，返回版本化结构化事实；自由文本只作审计。
- 预算记录增加 `trigger_reason`、`owner_strategy`、`model_role` 和
  `evidence_manifest_hash`，保证请求用途和归属可核对。
- `/api/status` 增加平均批次大小、跨策略复用数、缓存命中率、每股输入/输出 token、部分
  成功率、局部修复次数、Pro/emergency 原因和各软上限使用率。
- 普通推荐 API 不暴露 prompt、完整外部响应、内部证据载荷或模型自由文本。
- 本计划不增加历史下载任务、历史验证表、自动调参接口或全市场分钟线持久化。

## 8. 独立实施批次

### 8.1 long 彻底本地化

- 应用层令 long 的复核集合永久为空，不调用 `DeepSeekReviewPort`。
- 预算层拒绝 long 的主审、Pro、预热和 emergency。
- long 保持当前快照、本地评分、不冻结、不产生动作、不写历史。
- 更新权威文档、失败先行测试和 `CHANGELOG.md`，完成独立 Review、提交和推送后停止。

### 8.2 V4 事实提取与新闻处理

- 先更新权威策略、schema 和失败测试，再修改实现。
- 实现新闻公告去重、来源质量、证据冲突、催化状态和结构化风险事实。
- 实现结构化事实到维度分及风险处罚的本地纯函数映射。
- 保持融合公式、`ROUND_HALF_UP` 和固定验收向量 `83.40`。
- 旧冻结继续按原策略、引擎、prompt 和 schema 版本回放。

### 8.3 高价值路由与成本控制

- 实现请求时间窗口、三个短策略联合去重、候选优先级和正常软上限。
- 实现跨策略事实共享、自适应批处理、逐股接受和局部修复。
- 实现 Flash/Pro/emergency 的精确触发、原子计数和状态指标。
- DeepSeek 全部失败时仍须发布完整本地推荐并保留最近有效快照。

### 8.4 延期：收益验证与生产晋级

- 当前不下载或重建过去 60 日全市场数据。
- 后续由用户单独确认后，再建设向前收集的至少 60 个交易日、300 条有效配对和交易日
  分块 Bootstrap。
- 完成验证前，新评分方案不得替换活动策略版本，只能描述结构、成本、证据质量和可靠性
  改善，不能宣称提高收益。
- 只有净超额收益差的 95% 置信下界为正、严重回撤率下降且所有质量门禁通过，才允许晋级。

## 9. 测试与验收

- long 在所有阶段和预算桶下均产生 0 次 DeepSeek 请求。
- 共享预热只能在规定时间运行，09:30 后不补用额度。
- today 在 11:18、tomorrow/d25 在 14:48 后不得提交。
- 同一股票在三个短策略中只产生一次原始事实请求。
- tomorrow/d25 共享请求按固定规则记账，d25 不重复调用。
- 软上限耗尽后普通请求停止，未使用额度不跨策略转移。
- emergency 只能处理新重大风险或冻结保护集合变化。
- 官方确认且未充分反映的利好可进入映射；传闻和软新闻不能加分。
- 同一事件不会被本地新闻和 DeepSeek 重复加分。
- 单股 schema 错误不破坏同批合法结果，只修复错误子集。
- Pro 不能解除硬过滤、下行保护或新增无证据利好。
- 并发、重启、失败、超时、重试和修复始终受 188 次原子上限约束。
- DeepSeek 全部失败时仍产生完整本地推荐，保留最近有效快照并显式降级。
- 冻结不可变、旧版本回放、哈希一致性、融合 `83.40` 和三档桌面验收继续通过。

每个实施批次分别运行 Ruff、mypy、相关测试、完整测试、构建及仓库外 wheel 安装验收，
并按 `AGENTS.md` 独立 Review、提交和推送。

## 10. 假设与剩余风险

- 当前优先提升 DeepSeek 单次调用的信息价值并降低成本，不实施历史回测、自动调权、在线
  学习或自动晋级。
- today、tomorrow 和 d25 可以调用 DeepSeek；long 永久保持纯本地。
- 256 MiB 内存目标暂不调整；长期数据不常驻内存，成本优化主要依靠选择性调用、批处理和
  缓存。
- 正常软上限未来只能依据真实调用率和点时验证结果，在新的独立任务中修订，并同步更新
  权威策略和预算测试。
- 该计划尚未实施；当前活动代码和权威策略仍按现有版本运行，不能把计划值描述为已生效。
