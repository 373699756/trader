# 荐股全链路优化计划

状态：非权威执行计划，仅记录待实施批次与验收顺序。

产品、架构、时间线、API、运维和验收契约以
[软件业务设计文档](software-business-design.md) 为唯一权威；候选、过滤、公式、风险、
融合和排名策略以 [荐股策略文档](recommendation-strategy.md) 为唯一权威。本文不得覆盖
两份权威文档；实施批次若改变业务契约，必须先同步对应权威文档和测试。

## 1. 结论

从 SDK/API 获取股票数据，经标准化、硬过滤、评分，再到 Web 展示荐股信息的完整链条
仍有明显优化空间。当前优先级不是继续增加因子或提高 DeepSeek 权重，而是：

1. 修正历史复权语义；
2. 完成 P1-P6 性能与 Web 热路径；
3. 扩大候选总体的有效覆盖；
4. 建立可证明收益变化的同期影子验证。

本机固定负载单次测量如下；这些数字只用于定位，不替代每批固定 fixture 的重复门禁：

| 环节 | P95 | 门禁 | 结论 |
|---|---:|---:|---|
| 5,500 行标准化 | 217ms | 800ms | 通过 |
| 两源合并 | 1,020ms | 1,000ms | 略超 |
| 三源统一快照 | 1,616ms | 1,500ms | 超标 |
| 三板三策略评分 | 494ms | 1,000ms | 通过 |

Profile 显示，规范 JSON 序列化和哈希是合并热路径的主要成本；评分线程版的速度只有串行
版本约 0.78 倍，因此线程目前只应承担三板故障隔离，不能宣称计算加速。

本地参考仓库基线为 Lean `0269115d3cfb`、OpenBB `ebee248ed0bc`、TradingAgents
`a33fd4c0f134` 和 daily_stock_analysis `e685751f162f`。项目已经基本覆盖 OpenBB 的
provider 标准模型，以及 TradingAgents 的点时验证和显式 fallback；后续主要吸收 Lean
的复权来源建模、公司行为隔离、时间前沿和成本敏感性，以及 daily_stock_analysis 的
结构化结果归因。只参考机制，不复制外部源码，也不引入这些大型运行依赖。

## 2. 分批实施

### 2.1 批次 0：闭合当前 v17 任务

- 当前工作树已有约 50 个文件的 v17 下行保护、入场质量、结果结算和 Web 改动；先完成
  完整 Review、全门禁、CHANGELOG、单提交和推送。
- 不把后续性能或策略优化混入该提交。

### 2.2 批次 1：历史数据正确性

- 为历史序列增加明确的 `raw/qfq`、来源、截至日期和公司行为状态；需要前复权的均线、
  收益、ATR、波动和回撤拒绝消费未复权序列。
- 120 积分 Tushare `daily` 仅作为未复权行情、成交量和成交额来源；腾讯 qfq 作为价格类
  历史因子的主序列。对真实 30 代码批量调用增加集成验证，不能只依赖模拟 SDK 测试。
- 当前实时未复权价与历史 qfq 高点、均线比较时使用同一观察日缩放锚点；无法建立可验证
  锚点则保持缺失并降为观察。
- 增加分红、送转、拆并股、复权切换、跨源不一致和未来数据隔离回归测试，并提升策略与
  特征 schema 版本。

### 2.3 批次 2：完整交付 P1-P6 性能章节

- 按[六阶段性能契约](software-business-design.md#7-性能与资源模型)一次性交付六池缓存、
  `PublishedSnapshotIndex`、冻结检查点、驻留历史、冷读 single-flight、慢 SSE 客户端处理
  和 `perf-check` CLI。
- 普通盘中发布仅原子替换 P6 并并发 SSE，不再调用仓储写完整 SQLite/JSON；只有检查点、
  正式冻结和收盘 overlay 持久化。
- 优化 `src/trader/infra/market_data/merge.py` 的统一行情合并：使用显式原始类型投影、单次
  规范序列化和摘要复用；保持现有字段归属、排序、固定哈希向量及乱序确定性不变。
- 评分线程保持三板故障隔离；若基准加速比仍不大于 1，不增加更多线程。
- 退出条件为全部绝对预算通过、相同身份基线不退化超过 5%、100 tick 分配增长不超过
  20%。

### 2.4 批次 3：扩大候选总体，先影子后晋级

- 当前只预热约 360 只历史，约等于三板各自 120 的最终候选上限，存在上游隐式截断；
  建立覆盖全部支持 A 股的紧凑 `HistoryProfile` 点时库，完整 90 日序列仍只保留候选热集。
- 盘后以低优先级、可续跑方式补齐 qfq 历史；盘中只读取紧凑概要，不让历史抓取阻塞
  全市场行情。
- 新候选总体先作为 shadow：记录旧 360 总体与扩大总体的候选召回、行业覆盖、换手、
  分数分布和未来结果，不直接替换生产推荐。
- 只有点时覆盖率、样本外净超额收益和严重回撤同时满足既定门禁，才另开策略版本晋级；
  否则保留现有总体。

### 2.5 批次 4：建立收益证明闭环

- 复用同一份特征、风险事实和 DeepSeek 结果，同时生成 v17 与 `legacy_v16` 影子结果；
  不得增加 DeepSeek HTTP 请求，不写普通 Web。
- 使用独立 shadow 表保存生产快照引用、候选、动作、排名和结果，避免改变既有冻结表
  唯一键与历史哈希。
- 扩展结果结算：区分信号收益、20bp 主口径净超额收益、50/100bp 压力口径、MAE/ATR、
  停牌和价格限制导致的不可执行状态。
- 按交易日进行配对 block bootstrap，固定种子和 10,000 次重采样；输出策略/周期样本数、
  均值、中位数、胜率、严重回撤率及 95% 置信区间。
- 满足至少 60 个交易日、300 条有效配对样本、净超额收益差置信下界为正且严重回撤率下降
  后，才能声称收益改善；运行时不自动调权或回退。

## 3. 验证

- 每批先改权威契约和测试，再改实现，独立 Review、CHANGELOG、提交和推送。
- 运行 `make format-check`、`make lint`、`make type-check`、`make test`、`make package`，并在
  仓库外安装 wheel 验证导入、两个 CLI 和 Web 资源读取。
- 固定验证融合结果 `83.40`、预算并发、冻结竞态、哈希一致性、P6 无盘中完整写盘、
  Web/ETag/SSE 热路径和三档桌面分辨率。
- 策略变化额外执行点时防未来数据、公司行为、停牌/涨跌停、成本压力、因子消融、相关性
  和样本外稳定性测试。

## 4. 假设与边界

- 继续使用 Tushare 120 积分档，不购买更高积分能力。
- 不引入 OpenBB、Lean、TradingAgents 或 daily_stock_analysis 为运行依赖。
- 融合公式、188 次预算、冻结时间、只读 Web 和无真实下单边界保持不变。
- 在完成配对前瞻验证前，只能宣称正确性、覆盖率或性能改善，不能宣称荐股收益提高。
