# 无荐股与实时链路降级系统修复计划

状态：已实施、Review 并通过验收

基线：`2a801ed794d991ff34e63967bc71d823f988ae5b`

## 1. 用户可见问题

2026-07-24 盘中 Web 同时出现：

- tomorrow、d25 有数百只候选，但最终推荐为 0；
- `TopK live overlay degraded: market-data result completed after its batch deadline`；
- `board_data_reliability_below_threshold`、`deepseek_incomplete` 和
  `tomorrow_tail_data_incomplete`；
- 页面把所有空结果笼统描述为“未通过下行保护”，无法判断实际是无候选、分数不足、
  风险/执行限制还是集中度限制。

运行证据显示，历史预热在失败后立即递归重提同一批代码，计划/失败计数由约 359 万在
约五分钟内增长到约 459 万；候选仍有 tomorrow 223、d25 239，因此无荐股不是候选或
行业限制提前清空。TopK overlay 只刷新已经入选的股票，也不是短线空结果的直接原因。

## 2. Review 结论与边界

本计划不降低 today/tomorrow/d25 的正式或观察分数门槛，不强制凑股票，不改变固定
68/32 融合公式、风险扣分、行业最多 2 只的末位限制、冻结时间或冻结不可覆盖规则。

原方案经 Review 后作两项收敛：

1. DeepSeek 每日物理 HTTP 全局硬上限由 188 调整为 168。long 永久禁止物理调用，
   因此删除原有 18 个 long 兼容硬槽，再从 today 主阶段收回 2 次：

   | 桶 | today | tomorrow | d25 | long | shared_preheat | emergency | 合计 |
   | --- | ---: | ---: | ---: | ---: | ---: | ---: | ---: |
   | 硬上限 | 68 | 45 | 35 | 0 | 15 | 5 | 168 |

   阶段硬上限对应为 shared 15、today 15/40/13、tomorrow 25/20、d25 22/13、
   long 0、emergency 5；阶段目标为 shared 15、today 14/40/12、tomorrow 21/14、
   d25 19/11、long 0、emergency 0，合计 146。失败、超时、schema 修复和重试继续计数；
   shared preheat 与 emergency 保持独立 SQLite 原子预算桶。
2. 本批不新增历史协调线程，也不直接迁移 history source lane。先通过逐股负缓存退避
   终止无界链式重试，并验证实时池恢复；只有压力证据仍显示资源互相阻塞时，才另立任务
   调整线程拓扑。

## 3. 实施任务

### 3.1 历史预热负缓存与退避

- 失败代码按 60/120/240/480/900 秒逐级冷却，成功后清除失败状态。
- 当前批次失败后继续处理尚未尝试且不在冷却期的代码，不得立即重提相同失败批次。
- universe 变化时清理已移除代码的失败状态，避免内存无界增长。
- 修复 warmup 锁与 history 锁的潜在反向获取；读取历史 entries 不得持有 warmup 锁。
- 状态新增冷却代码数、下一次重试秒数和唯一失败代码数；累计计划/完成/失败仍保留。

### 3.2 尾盘数据板块公平优先

- 保持候选集合、候选分和返回顺序不变。
- 仅在 intraday I/O 内部，把已经按候选分排序的代码按主板、创业板、科创板稳定轮询，
  使 3 秒预算优先覆盖三板高分候选，而不是只覆盖全局排序最前板块。
- 不用尾盘分反向决定自身抓取顺序，不改变评分公式或冻结回放输入。

### 3.3 有界 DeepSeek 复核集合

- 每策略、每次投影最多复核 24 只。
- 优先级为新高风险、距动作门槛 5 分内、全局 TopK 边界、证据冲突、本地排名；
  可靠度低于 0.85 的股票仍不调用。
- 混合模式保护集合与实际待审集合使用同一个确定性选择函数，避免“未计划复核的股票”
  反过来令整个快照永久 `local_degraded`。
- v18 及更早冻结快照继续使用旧待审范围回放；活动引擎升级为 v19。
- 未处理候选继续使用本地评分，不因模型预算或网络失败而退出推荐链。

### 3.4 空结果选择诊断

每个新快照确定性记录并通过推荐 API/状态 API 投影：

- `scored_candidate_count`
- `actionable_candidate_count`
- `score_qualified_count`
- `selection_floor`
- `maximum_local_score`
- `maximum_final_score`
- `empty_reason`

`empty_reason` 固定区分 `no_scored_candidates`、`score_below_observation_floor`、
`risk_or_execution_blocked` 和 `selection_limits`。页面空表根据该字段显示真实原因；
例如“最高评分 69.42，低于观察门槛 73.00，本轮不荐股”。数据降级原因继续单独展示，
不得和正常分数不足混为一谈。

## 4. 验证与上线

- 单元/组件回归证明永久失败代码不会形成紧密循环，冷却到期后按级数重试，其他代码
  能继续完成。
- 168 并发原子预算探针必须只允许 168 次，其余统一以 `daily_hard_limit` 拒绝；long
  在所有入口保持 0 次。
- 三板 intraday 优先顺序、待审集合 24 只上限、旧冻结回放和新选择诊断均需确定性测试。
- 运行完整 format、Ruff、mypy、pytest、package、仓库外 wheel 安装与三档桌面浏览器验收。
- 修复代码验证通过后再优雅停止当前服务并以单实例重启；保留 runtime SQLite、冻结记录
  和检查点。暂停的旧进程只在明确授权后终止。
- 上线后观察至少 10-15 分钟：历史计划数不得再无界增长，冷却代码不重复请求，实时队列
  和 TopK 年龄恢复到契约范围。即使最终仍为 0，也必须能展示最高分、门槛和真实空结果原因。

实机验收于 2026-07-24 完成：新进程在 14:52 至 15:04 的观察窗口内，历史预热由计划
360/完成 358/失败 2 有界推进到计划 421/完成 414/失败 7，冷却 5、唯一失败 7，未再出现
百万级自旋；15:00 后 today/tomorrow/d25 分别以 133/220/238 个候选创建
`close_fallback`，每策略发布 2 只观察推荐。随后重启进程，从不可变冻结记录恢复三个
当前视图，推荐 API 均返回 2 项及完整 `selection_diagnostics`，状态为 `running`、
`phase=after_close`、`last_error` 为空。
