# A 股荐股研究系统完整实施需求

版本：v6
状态：实施基线  
用途：本文是精简重写版程序的唯一业务与工程需求契约。

## 1. 建设目标

在当前开发分支直接重写精简版程序，提供：

- `today_plan`：09:36-11:20 盘中短线研究信号。
- `tomorrow_plan`：T 日尾盘形成、面向 T+1 的推荐。
- `d25_plan`：T 日尾盘形成、面向 T+2 至 T+5 的推荐。
- `long_plan`：固定长期观察池，只展示，不产生动作、不持久化推荐历史。
- 实时行情、硬过滤、三策略评分、DeepSeek 五维复核、风险合并、TopK、冻结、历史锚点和统一 Web 看板。

系统允许返回 0 只推荐，不连接券商、不执行真实交易。所有页面保留“仅供研究，不构成投资建议”。

删除策略验证、收益回填、回测工作台、自动调参、股票预测及相关页面和 API。离线点时检验仅作为开发质量门禁，不作为产品功能。

## 2. 开源项目参考

只借鉴经过验证的机制，不直接复制收益参数：

- Qlib：因子组织、点时数据、时间切分。
- RQAlpha：A 股交易日历、停牌和涨跌停规则。
- vn.py、WonderTrader：事件调度、任务生命周期。
- QUANTAXIS：行情适配和数据标准化。
- CZSC：多周期信号表达。

GitHub star 数不代表策略收益。首版不直接依赖上述项目源码；如采用源码，必须固定 commit、核实许可证并记录归属。公开策略参数必须重新通过 A 股点时数据、停牌、涨跌停、滑点和样本外检查。

四个边界的职责和依赖规则如下：

- `domain`：领域数据类、字段标准化后的纯计算、硬过滤、三策略评分、风险和 TopK。允许标准库及确定性数值计算依赖，不得读取配置文件、环境变量、时钟、网络或数据库。
- `application`：定义外部端口、事件、调度、推荐编排、发布和冻结用例，只依赖 `domain`。
- `infrastructure`：实现行情、交易日历、DeepSeek、SQLite、JSON、缓存和配置端口，可以依赖 `application` 和 `domain`。
- `web`：只负责 Flask 路由、请求校验、响应序列化、SSE 和页面资源，只通过 `application` 用例读取已发布快照。
- `bootstrap.py` 是唯一组合根，负责创建具体适配器和注入依赖；其他模块不得使用全局服务定位器或在函数内部隐藏创建网络/数据库客户端。
- `entrypoints` 只处理进程参数、生命周期和退出码。`create_app()` 位于 `web/app.py`，仍保持无网络、无线程、无数据库副作用。

公开进程入口固定为 `trader-server` 和 `trader-cli`，由 `pyproject.toml` 声明；跨平台 run 脚本调用这些入口并保持现有一键启动方式。外部配置必须通过 `--config` 或 `TRADER_CONFIG` 传入绝对路径，run 脚本默认指向仓库 `config/v2/runtime.json`，任何模块不得按当前工作目录猜测配置位置。wheel 安装测试使用显式临时配置路径。

依赖方向固定为 `entrypoints/web/infrastructure -> application -> domain`。新增架构契约测试扫描导入关系；任何反向导入、跨层访问适配器内部对象或新建含义模糊的 `utils.py`、`common.py` 都视为失败。单文件原则上不超过 500 行；超过时必须按用例或适配器职责拆分并在评审中说明原因，禁止再形成聚合全部业务的 `app_services.py` 类模块。

根目录只保留项目元数据、跨平台启动脚本和一级工程目录。HTML/CSS/JavaScript 移入 `src/trader/web` 并作为包数据构建；`pyproject.toml` 是依赖和包发现的唯一权威，重复的根 `requirements.txt` 不再手工维护，需要可重复安装时只保留由权威声明生成的约束文件。`build/`、`dist/`、`*.egg-info`、缓存和覆盖率文件均为生成物，不进入 Git。旧研究、验证和预测文档移入 `docs/archive/v1`，不再与 `need.md` 竞争业务契约。

旧 `stock_analyzer` 和旧 `.runtime` 在迁移期只用于回归对照，不允许被 `src/trader` 导入。新版使用独立 `.runtime/v2`；验证完成后原子切换启动入口，再删除活动代码中的旧包，旧运行数据始终保留。

## 4. 生命周期与多线程流水线

`create_app()` 只创建 Flask 应用，不创建线程、不访问网络。正式入口按顺序加载配置、创建适配器、创建运行时、启动流水线、启动 Web；退出时停止接收事件、完成冻结写入、关闭线程池。

```text
调度线程
  |- 全市场行情事件 -|
  |- 候选报价事件   |-> 标准化 -> 硬过滤 -> 特征快照
  `- 新闻风险事件 --'                    |
                         |---------------|---------------|
                         v               v               v
                     today评分       tomorrow评分       d25评分
                         `---------------|---------------'
                                         v
                              DeepSeek 五维批量复核
                                         v
                               veto/风险合并 -> TopK
                                         v
                            单写线程 -> SQLite + JSON
                                         v
                                  Web/SSE 只读发布
```

默认线程：调度 1、数据采集 6、标准化/过滤 2、策略评分 3、DeepSeek 4、合并 1、持久化 1。所有线程池和队列有界，不按股票或事件无限创建线程。

long 使用低优先级独立展示分支，复用标准化、风险和 DeepSeek 原始五维结果，但不占用三策略评分 worker、不进入冻结写入。

事件包含 `event_id`、类型、主体键、交易日、阶段、策略、优先级、行情版本、配置版本、创建时间、截止时间、重试次数和载荷引用。幂等键为 `trade_date + phase + strategy + event_type + subject_key + data_version`；市场级主体使用固定值 `market`，个股级主体使用规范化六位代码。事件状态转换采用 compare-and-set，同一幂等键只能有一个有效执行者。

事件优先级：冻结 > 风险变化 > DeepSeek 补审 > 策略评分 > 候选行情 > 全市场行情 > 长期数据。普通行情队列满时合并同主体旧版本并保留最新版本；风险和冻结事件使用独立保留容量并先写入运行库，进程重启后重放，不能只依赖内存队列。队列容量、合并数、拒绝数和重放数必须进入状态指标。

## 5. 交易时间线

- 09:15-09:30：共享预热，构建候选、历史因子和证据。
- 09:30-09:36：today 观察，不产生可执行动作。
- 09:36-10:30：today 主执行。
- 10:30-11:20：today 降级执行，提高阈值。
- 11:20：冻结 today 最终信号；此后不新增可执行推荐，只更新展示报价。
- 13:00-14:20：tomorrow、d25、long 预计算。
- 14:20-14:48：新入围、风险变化和证据变化补审。
- 14:48：停止提交生产 DeepSeek 请求。
- 14:49:50：强制刷新最终候选报价。
- 14:50：冻结 tomorrow、d25 及当日最终展示快照；long 只发布当前观察快照，不形成历史推荐。
- 15:00：保存收盘行情，只更新展示字段，不修改冻结策略。

所有业务时间使用 `Asia/Shanghai`。窗口统一采用左闭右开区间，单点任务以 `HH:MM:SS` 精确触发，文档中的“截止”表示该时刻起禁止新提交。交易日以 A 股交易日历判断，不能只用工作日代替；日历源失败时使用带版本和更新时间的本地缓存，缓存缺失或过期不得猜测交易日，只能进入降级只读状态。

## 7. 实时性、限流与降级

每条数据保存来源时间、接收时间、发布版本和来源名称；乱序结果不得覆盖新版本。

实时目标：

- TopK 价格：关键阶段 P95 年龄不超过 10 秒，其他交易阶段不超过 20 秒。
- 候选报价：主执行不超过 15 秒，其他阶段不超过 30 秒。
- 全市场快照：主执行不超过 60 秒，其他阶段不超过 120 秒。
- SSE 发布延迟 P95 不超过 2 秒。
- today 行情变化到评分发布 P95 不超过 15 秒。
- 11:20 today 锚点报价年龄不超过 20 秒，目标不超过 10 秒。
- 14:50 锚点报价年龄不超过 30 秒，目标不超过 10 秒。

数据年龄超过计划周期 2 倍标记 `stale`，超过 3 倍进入 `degraded`。today 行情超过 20 秒、tomorrow/d25 冻结行情超过 30 秒，只能观察。

全市场请求截止 20 秒、候选报价 3 秒、新闻事件 8 秒、DeepSeek 默认 12 秒。同一来源 single-flight；连续失败 3 次熔断 60 秒，半开时只允许一个探测请求。

网络失败不得清空推荐。优先展示最近有效快照，并明确数据来源、年龄和降级原因。

## 8. 候选池与字段标准化

预选分：

`candidate_score = 35%流动性 + 25%短期动量 + 20%趋势 + 10%行业强度 + 10%数据完整度`

通过硬过滤后取前 120 只。DeepSeek 不能新增池外股票。

指标按横截面 2.5%/97.5% 分位截尾，再映射到 0-100。缺失值用中性分 50，但必须记录 `missing_fields`；核心字段缺失超过 30% 只能观察。

统一字段至少包括价格、涨幅、5 分钟变化、涨速、量比、换手、成交额、振幅、MA5/20/60 偏离、3/5/10/20/60 日收益、波动率、最大回撤、行业强度、新闻情绪、风险标记和数据时间。

所有因子必须登记在受版本控制的 `config/v2/strategy.json`，字段至少包括 `factor_id`、适用策略、原始输入、精确公式、单位、方向、观察时点、复权方式、回看窗口、最小样本数、截尾和归一化规则、缺失策略、输出范围及版本。未登记因子不得进入生产评分。

统一计算约定：实时价格和冻结锚点使用未复权价格；历史收益、均线和回撤使用在观察时点可获得的前复权序列；百分比字段以百分点表示，`1.2` 表示 `1.2%`；横截面分位的总体为同一 `trade_date + phase + data_version` 下通过代码、交易状态和数据有效性基础检查的股票，重复值取平均秩；NaN、无穷值和样本不足均按缺失处理，不得静默转换为 0。因子配置、风险规则、阈值和融合公式共同生成不可变 `strategy_version` 与 `config_version`。

## 9. 硬过滤规则

命中任一条件立即剔除，不计算任何分数：

- 非法沪深主板、创业板、科创板代码。
- ST、退市风险、停牌、无价格、无有效成交。
- 行情超出新鲜度或多源价格偏差大于 0.5% 且复核失败。
- 20 日成交额中位数低于 5000 万元。
- 一字涨跌停、明显异常报价、黑名单、重大退市或监管风险。
- 主板当日涨幅大于 8%。
- 创业板/科创板当日涨幅大于 16%。

边界必须精确测试：主板 8.00% 通过、8.01% 剔除；创业板/科创板 16.00% 通过、16.01% 剔除。

过滤结果保存 `filter_code`、阈值、实际值、来源和时间，不允许只返回总布尔值。

## 10. 本地评分基础公式

定义 `B(x;l,o1,o2,u)`，参数必须满足 `l < o1 <= o2 < u`：`x <= l` 或 `x >= u` 得 0，`o1 <= x <= o2` 得 100，`l < x < o1` 时为 `100 * (x-l)/(o1-l)`，`o2 < x < u` 时为 `100 * (u-x)/(u-o2)`。无效参数使配置启动校验失败，缺失或非有限 `x` 走因子缺失规则。`P(x)` 为 0-100 横截面平均秩分，方向相反的因子必须在因子登记中显式声明并执行 `100-P(x)`。

```text
base_score = sum(component_score * component_weight)
local_score = clamp(base_score - local_risk_penalty, 0, 100)
```

组件和子组件权重都必须精确合计为 1.0，启动时校验；计算过程保留完整精度，统一舍入规则见第 17 节。

本地风险扣分上限 25：

- 近涨停拥挤 5。
- 量价背离 4。
- 高波动 3。
- 减持/解禁 3-8。
- 质押风险 3-8。
- 财务恶化 6。
- 负面公告或监管风险 5-15。

每个风险事实必须生成稳定 `risk_fact_id`，同一事实在本地规则中只能扣一次，必须展示风险代码、实际值、阈值、来源、证据时间和扣分。

`config/v2/strategy.json` 必须把上述区间规则展开为确定的风险表，每项登记 `risk_code`、适用策略、触发公式、阈值、严重度、扣分、最低置信度、证据有效期、互斥/叠加组和 `risk_fact_id` 生成字段。所有可变区间在配置中必须落到精确分段，不能由实现自行选择 3-8 或 5-15 内的数值；本地合计超过 25、DeepSeek 合计超过 30 时分别截断并保留截断前明细。

## 11. today_plan 评分

组件权重：动量 35%、流动性 25%、行业 10%、情绪 20%、风险保护 10%。

动量：

- 5 分钟涨幅 `B(0,0.2,1.8,3.5)`：30%。
- 涨速分位：20%。
- 当日涨幅 `B(-1,1,5.5,8)`：20%。
- 量比 `B(0.8,1.2,3.5,6)`：15%。
- 3 日相对强度：15%。

流动性为 20 日成交额分位 60% + 换手率 `B(0.5,1.5,8,15)` 40%。行业为行业强度 70% + 行业上涨宽度 30%。情绪为新闻极性 50% + 证据新鲜度 30% + 市场宽度 20%。保护为低波动 35% + 低回撤 35% + 低拥挤 30%。

## 12. tomorrow_plan 评分

权重：流动性 25%、动量 15%、趋势 20%、历史优势 15%、执行 10%、尾盘结构 15%。

- 动量：5 日强度 35% + 20 日强度 25% + 量价确认 25% + 当日涨幅适中 15%。
- 趋势：MA20/60 位置 30% + 均线斜率 30% + 20 日突破 20% + 行业趋势 20%。
- 历史优势：20 日波动调整收益 45% + 低最大回撤 30% + 上涨一致性 25%。
- 执行：成交容量 40% + 振幅适中 30% + 距涨停安全度 30%。
- 尾盘：最后 30 分钟收益 35% + 尾盘量比 30% + 收盘位置 35%。

尾盘分钟数据只对候选池抓取，不对全市场逐股请求。

## 13. d25_plan 与 long_plan

d25 权重：动量 30%、趋势 25%、流动性 20%、执行 15%、不过热 10%。

- 动量：5/10/20 日强度 30%/30%/25% + 量价确认 15%。
- 趋势：MA20/60 结构 35% + 斜率 30% + 突破 20% + 行业趋势 15%。
- 执行：成交容量 40% + 振幅 30% + 价格可执行性 30%。
- 不过热：MA20 偏离反向 40% + 20 日涨幅不过热 35% + 低波动 25%。

20 日涨幅不高于 15% 时过热系数 1.0，15%-30% 线性降至 0.85，高于 30% 为 0.75。市场状态系数为风险偏好 1.03、中性 1.0、风险规避 0.92。d25 固定计算顺序为 `adjusted_base_score = clamp(base_score * overheat_factor * market_regime_factor, 0, 100)`，再执行 `local_score = clamp(adjusted_base_score - local_risk_penalty, 0, 100)`；市场状态分类公式必须登记在因子配置中，不允许运行时人工切换。

long 使用受版本控制的固定名单，按价值 30%、成长 25%、质量 20%、行业/政策 15%、风险保护 10% 排序。目标价为可选人工配置，未配置显示 `--`，不得由模型虚构。long 不产生执行动作、不写推荐历史、不参与日期查询；允许保存通用 DeepSeek 调用审计和短期缓存，但不得把它们解释为长期推荐记录。

## 14. DeepSeek 当前问题与改造目标

旧系统存在分析批次为零、调度租约长期停留 `running`、定性证据门控提前过滤全部候选、子进程状态不可见等问题。

新版取消调度子进程，调度线程直接提交有界 DeepSeek 任务。候选只要通过硬过滤且具有行情和至少一组结构化特征即可调用；新闻、公告不是调用资格。

缺少单类证据时仅将该维度设为 `unknown`，不能因此把整只股票提前 `abstain`。批次任务必须进入 `success`、`partial`、`failed`、`skipped` 或 `abandoned` 终态；每只股票另有 `applied`、`abstain`、`rejected` 或 `late` 结果，不能混用批次状态与候选结果。全部维度未知或加权置信覆盖不足时，候选结果为 `abstain` 并回退本地分。

## 15. DeepSeek 五维规则

五维为价值质量、财务健康、资金量价、行业政策、风险质量。每维输出：

- `score`：0-100。
- `confidence`：0-1。
- `flags`。
- `evidence_ids`。
- `assessment`。

每个维度的原始 `score` 和 `confidence` 必须是有限数并分别位于 0-100、0-1；维度有效分为 `effective_i = 50 + (score_i - 50) * confidence_i`，未知维度固定为 `score=50, confidence=0, effective=50`，已知维度权重不因其他维度未知而重新归一化。

| 策略 | 价值质量 | 财务健康 | 资金量价 | 行业政策 | 风险质量 |
| --- | ---: | ---: | ---: | ---: | ---: |
| today | 10% | 10% | 40% | 15% | 25% |
| tomorrow | 15% | 20% | 25% | 20% | 20% |
| d25 | 20% | 25% | 20% | 20% | 15% |
| long | 30% | 30% | 10% | 20% | 10% |

对当前策略使用表中权重 `w_i`：

```text
deepseek_score = sum(effective_i * w_i)
confidence_coverage = sum(confidence_i * w_i)
```

仅当 schema 完整、证据引用有效、结果在阶段截止前到达、候选非 `abstain`、至少两个维度已知且 `confidence_coverage >= 0.50` 时，`deepseek_applied=true`。阈值是策略契约的一部分；后续修改必须提升策略版本并通过离线点时检验。`partial` 批次中的单只股票只要独立满足上述条件仍可应用，其余股票逐只回退。

DeepSeek 不能新增股票。池外代码、无效证据引用和无法解析的 schema 结果直接拒绝。

风险维度除通用字段外，返回结构化 `risk_facts`，每项包含 `risk_code`、`severity`、`confidence`、`evidence_ids` 和 `assessment`。模型不得直接决定生产扣分；本地规则根据已验证事实映射 `deepseek_risk_penalty`，范围为 0-30。与本地风险集合具有相同 `risk_fact_id` 的事实不再进入 DeepSeek 数值扣分，避免重复处罚；连续风险质量分、离散风险扣分和 veto 必须分别展示。

高风险证据有效且置信度不低于 0.7 时才允许 `deepseek_veto=true`；模型输出不得直接决定 veto，本地风险表必须显式登记 veto 资格、允许证据类型和证据有效期后才能映射。缺数据、证据类型不匹配、证据过期、低置信度或单纯看空意见不得 veto。数值扣分同样必须有池内证据引用并达到对应风险规则的最低置信度，veto 可以在数值去重后继续作为独立安全门生效。

单批最多重试一次；429 遵循 `Retry-After`。14:48 后不提交请求；迟到结果只保存审计，不参与生产。

## 17. 合并、动作与 TopK

融合公式版本固定为 `fusion_v2_local68_deepseek32`。`local_score` 已按第 10 节扣除本地风险，最终融合不得再次扣除 `local_risk_penalty`。候选级 `fusion_applied = deepseek_applied and fusion_mode == "hybrid"`：

```text
if fusion_applied:
    raw_final_score = local_score * 0.68 + deepseek_score * 0.32 - deepseek_risk_penalty
else:
    raw_final_score = local_score

final_score = round_half_up(clamp(raw_final_score, 0, 100), 2)
```

`deepseek_risk_penalty` 是融合后的绝对分值，不乘以 0.32。DeepSeek 失败、弃权、超时、额度耗尽、迟到、覆盖不足或 schema 无效时，该候选 `final_score = round_half_up(local_score, 2)`、DeepSeek 扣分为 0、普通 DeepSeek veto 不生效；已经由证据规则验证的重大安全 veto 仍可独立阻止执行。排名、动作判断、持久化和 API 都使用同一个两位小数规范分，不允许界面舍入值与门槛判断值不同。

每个 `strategy + data_version` 只能发布一种 `fusion_mode`：

- `hybrid`：排名保护集合中的候选均已得到 `applied` 或 `abstain` 终态，`applied` 使用 68/32 公式，`abstain` 使用本地回退。
- `local_degraded`：保护集合中仍有失败、迟到、拒绝或未完成候选，整版排名统一使用本地分，避免未复核股票因回退而系统性占优；页面明确显示降级原因。

排名保护集合为本地分前 18 名与距当前策略动作门槛 5 分以内候选的并集。模式切换必须产生新发布版本，冻结时模式一并固化。

默认 TopK 为 10，接口允许 0-18；`top_n=0` 合法并返回空列表。排序固定为最终分降序、本地分降序、股票代码升序；单一行业最多 3 只，不足时不得降低门槛补满。

- today 主窗口：`final_score >= 70` 可执行。
- today 降级窗口：`final_score >= 76` 可执行。
- tomorrow：`final_score >= 72` 可执行。
- d25：`final_score >= 70` 可执行。
- 距门槛 5 分以内：观察。
- 其他情况：不可执行。

所有可执行项必须满足硬过滤、时间窗口、行情新鲜、无 veto 和周期匹配。

上述动作门槛是 v8 当前候选值。切换生产前必须使用冻结点时输入完成离线重放，并至少报告各策略分数分布、推荐数、空推荐比例、TopK 变化、DeepSeek 覆盖、`local_degraded` 比例和风险拦截率；门槛未通过预注册验收时必须在本文和 `strategy_version` 中同步修订，禁止运行时自动调参或只改配置不升版本。

## 18. 冻结与持久化

各策略冻结边界固定为：

| 策略 | 冻结时间 | 锚点要求 | 是否写推荐历史 |
| --- | --- | --- | ---: |
| today | 11:20 | 最近有效候选报价，年龄不超过 20 秒 | 是 |
| tomorrow | 14:50 | 14:49:50 最终报价，年龄不超过 30 秒 | 是 |
| d25 | 14:50 | 14:49:50 最终报价，年龄不超过 30 秒 | 是 |
| long | 不冻结 | 当前展示报价 | 否 |

冻结排名、评分、动作、veto、阈值、融合模式、策略版本、配置版本、数据版本和锚点。冻结后只允许在独立 `live_overlay` 更新最新价、今日涨跌和锚点至今，不得修改冻结记录或冻结 JSON。

推荐 SQLite 表 `recommendations` 严格保存：

`strategy`、`recommend_date`、`stock_code`、`rank`、`anchor_price`、`anchor_daily_return_pct`、`snapshot_id`

主键为 `(strategy, recommend_date, stock_code)`。

`frozen_snapshots` 保存 `snapshot_id`、策略、日期、冻结时间、融合/策略/配置/schema 版本、数据版本、JSON 相对路径、SHA-256、记录数和 `staged/committed` 状态，`(strategy, recommend_date)` 唯一。完整候选总体、标准化输入、缺失掩码、公式中间值、评分、风险事实、证据、DeepSeek 调用批次、动作、veto、过滤与降级原因保存到按 `strategy/date/snapshot_id.json` 命名的不可覆盖 JSON，保证能从冻结输入独立复算。

运行库另设：

- `pipeline_events`：事件状态、阶段、重试和终态。
- `deepseek_calls`：模型、批次、HTTP 状态、token、延迟和错误。
- `data_source_health`：来源成功率、延迟、熔断和数据年龄。

SQLite 与 JSON 只允许单写线程操作。实时草稿写入可替换的 `published/`；冻结时先在 SQLite 事务中以预计算路径和 SHA-256 创建 `staged` manifest，再写同目录临时文件、刷新并原子改名，最后在第二个 SQLite 事务中写推荐行、已发布指针并把 manifest 改为 `committed`。启动恢复扫描 `staged` manifest、孤儿文件、哈希不符和缺失文件：完整项幂等提交，不完整项隔离并回退最近有效快照，禁止静默覆盖。

## 19. Web API

公开接口：

- `GET /api/status`
- `GET /api/recommendations/<strategy>?date=YYYY-MM-DD&top_n=10`
- `GET /api/recommendation-dates?strategy=<strategy>`
- `GET /api/events?cursor=<cursor>&limit=100`
- `GET /api/events/stream`

`strategy` 只接受 `today`、`tomorrow`、`d25`、`long`；long 只允许当前日期或省略日期，不出现在历史日期接口。`top_n` 必须为 0-18 的整数，日期、游标或策略无效返回结构化 400，快照不存在返回 404，服务已启动但尚无快照返回 200 空结果并带 `not_ready` 状态。所有响应使用版本化 envelope，至少包含 `schema_version`、`snapshot_id`、`published_at`、`data_version`、`strategy_version`、`fusion_version`、`fusion_mode`、`stale`、`degraded_reasons`、`items` 和 `error`。

Web 处理器只读取已发布快照，不允许直接抓取行情、计算评分或调用 DeepSeek。当前快照支持 ETag；相同版本返回 304，避免轮询重复传输。

`/api/events` 是有界审计查询，`/api/events/stream` 是 SSE。SSE 只推送已发布版本，事件 `id` 使用单调发布序号，支持 `Last-Event-ID`；游标仍在保留窗口内时补发，过旧时发送 `resync_required` 并由前端获取完整快照。心跳 15 秒；断线后前端每 15 秒轮询，恢复 SSE 后停止轮询，避免双重刷新。服务端必须限制单客户端缓冲和总连接数，慢客户端不得阻塞发布线程。

## 24. 分阶段实施

目录优化不是一次性 `mv`。迁移期间旧应用可以继续从旧入口运行，新包不得导入旧包，同一进程也不得混合新旧调用链。每阶段通过门禁后才进入下一阶段。

### 24.0 “继续”命令的整节交付协议

用户每发送一次“继续”或语义等价的继续执行指令，都推进计划或需求文档中下一个完整未完成章节，以同级标题为章节边界，并完成章节内全部明确子项。同一章节的多个可独立验收子项必须作为一个整节任务统一实现、回归、Review、提交和推送，不得完成首个子项后提前停止，也不得顺带处理相邻章节或无关优化。直接下达的新任务同样形成独立交付批次；用户明确指定完整章节时按整节交付，否则多个互不相关目标仍拆成多个任务逐次交付。

每个整节任务固定执行以下闭环：

```text
记录已推送基线和已有工作树变更
  -> 选择下一完整未完成章节并限定文件范围
  -> 完成章节内全部明确子项与针对性验证
  -> 独立 Review
  -> 修复发现并重新 Review，直到零已知问题
  -> 更新 CHANGELOG.md
  -> 创建一个 Conventional Commit
  -> 推送当前跟踪分支并核对本地/上游提交一致
  -> 停止，等待下一次“继续”
```

本章节任务 Review 必须以上一次已推送提交为基线检查完整 diff，不能由阶段末或整体计划 Review 代替。Review 通过只结束审查计划项，章节内全部子项完成前交付任务仍保持进行中。每个整节任务只允许一个新提交和一次对应的成功推送记录，不得把多个“继续”批次或相邻章节合并、不得修改已推送任务历史。网络、权限或远端拒绝导致推送失败时，保留本地提交并明确报告阻塞；推送重试仍属于当前章节任务，直到本地 `HEAD` 与 `@{upstream}` 哈希一致后任务才算闭合，才能开始下一章节。

## 25. 最终验收标准

- 正常交易时段连续运行，TopK 报价 P95 年龄不超过 10 秒。
- `final_score` 严格使用 68% 本地分、32% DeepSeek 分并只额外扣除去重后的 DeepSeek 风险，固定验收向量得到 83.40。
- 普通结构化候选能够进入 DeepSeek，密钥有效且候选存在时调用不能为 0。
- DeepSeek 达到阶段目标，或逐项解释未达原因。
- 所有过滤、评分、风险、veto 和最终排名可从冻结输入复算。
- 任一数据源或 DeepSeek 失败不阻塞本地推荐和 Web。
- 11:20/14:50 后任何迟到事件都不能修改相应冻结结果，冻结文件和 SQLite manifest 哈希一致。
- UI 在目标桌面分辨率下风格统一、无白屏、无重叠、无页面级横向溢出和明显刷新抖动。
- 活动产品代码只存在于 `src/trader`，新包不导入旧包，根目录无活动业务模块、重复依赖声明或被跟踪的构建/缓存产物。
- `pyproject.toml` 能构建 sdist/wheel；在仓库外安装后 server、CLI、模板和静态资源可用。
- Ruff、mypy、pytest、架构契约、构建安装、并发测试和 UI 截图验收全部通过。
