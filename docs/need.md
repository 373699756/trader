# A 股荐股研究系统完整实施需求

版本：v6
状态：实施基线  
用途：本文是精简重写版程序的唯一业务与工程需求契约。

## 1. 建设目标

在当前开发分支直接重写精简版程序，提供：

- `today_plan`：09:36-11:20 盘中短线研究信号。面向 T+1 的推荐。
- `tomorrow_plan`：T 日尾盘形成、面向 T+1 的推荐。
- `d25_plan`：T 日尾盘形成、面向 T+2 至 T+5 的推荐。
- `long_plan`：固定长期观察池，只展示，不产生动作、不持久化推荐历史。
- 实时行情、硬过滤、三策略评分、DeepSeek 五维复核、风险合并、TopK、冻结、历史锚点和统一 Web 看板。

系统允许返回 0 只推荐，不连接券商、不执行真实交易。所有页面保留“仅供研究，不构成投资建议”。

删除策略验证、收益回填、回测工作台、自动调参、股票预测及相关页面和 API。离线点时检验仅作为开发质量门禁，不作为产品功能。

四个边界的职责和依赖规则如下：

- `domain`：领域数据类、字段标准化后的纯计算、硬过滤、三策略评分、风险和 TopK。允许标准库及确定性数值计算依赖，不得读取配置文件、环境变量、时钟、网络或数据库。
- `application`：定义外部端口、事件、调度、推荐编排、发布和冻结用例，只依赖 `domain`。
- `infrastructure`：实现行情、交易日历、DeepSeek、SQLite、JSON、缓存和配置端口，可以依赖 `application` 和 `domain`。
- `web`：只负责 Flask 路由、请求校验、响应序列化、SSE 和页面资源，只通过 `application` 用例读取已发布快照。
- `bootstrap.py` 是唯一组合根，负责创建具体适配器和注入依赖；其他模块不得使用全局服务定位器或在函数内部隐藏创建网络/数据库客户端。
- `entrypoints` 只处理进程参数、生命周期和退出码。`create_app()` 位于 `web/app.py`，仍保持无网络、无线程、无数据库副作用。

公开进程入口固定为 `trader-server` 和 `trader-cli`，由 `pyproject.toml` 声明；跨平台 run 脚本调用这些入口并保持现有一键启动方式。外部配置必须通过 `--config` 或 `TRADER_CONFIG` 传入绝对路径，run 脚本默认指向仓库 `config/v2/runtime.json`，任何模块不得按当前工作目录猜测配置位置。wheel 安装测试使用显式临时配置路径。

依赖方向固定为 `entrypoints/web/infrastructure -> application -> domain`。新增架构契约测试扫描导入关系；任何反向导入、跨层访问适配器内部对象或新建含义模糊的 `utils.py`、`common.py` 都视为失败。单文件原则上不超过 500 行；超过时必须按用例或适配器职责拆分并在评审中说明原因，禁止再形成聚合全部业务的 `app_services.py` 类模块。

根目录只保留项目元数据、跨平台启动脚本和一级工程目录。HTML/CSS/JavaScript 移入 `src/trader/web` 并作为包数据构建；`pyproject.toml` 是依赖和包发现的唯一权威，重复的根 `requirements.txt` 不再手工维护，需要可重复安装时只保留由权威声明生成的约束文件。`build/`、`dist/`、`*.egg-info`、缓存和覆盖率文件均为生成物，不进入 Git。旧研究、验证和预测文档移入 `docs/archive/v1`，不再与 `need.md` 竞争业务契约。

旧 `stock_analyzer` 和旧 `.runtime` 在迁移期只用于回归对照，不允许被 `src/trader` 导入。新版使用独立 `.runtime/v2`；验证完成后原子切换启动入口，再删除活动代码中的旧包，旧运行数据始终保留。

## 4. 生命周期与多线程流水线

`create_app()` 只创建 Flask 应用，不创建线程、不访问网络。正式入口按顺序加载配置、创建适配器、创建运行时、启动流水线、启动 Web；退出时停止接收事件、完成冻结写入、关闭线程池。

```text
调度线程
  |- 全市场行情事件 -|
  |- 候选报价事件   |-> 标准化 -> 硬过滤 -> 特征快照
  `- 新闻风险事件 --'                    |
                         |---------------|---------------|
                         v               v               v
                     today评分       tomorrow评分       d25评分
                         `---------------|---------------'
                                         v
                              DeepSeek 五维批量复核
                                         v
                               veto/风险合并 -> TopK
                                         v
                            单写线程 -> SQLite + JSON
                                         v
                                  Web/SSE 只读发布
```

默认线程：调度 1、数据采集 6（五来源普通 lane 5、候选/TopK 紧急 lane 1）、标准化/过滤 2、策略评分 3、DeepSeek 4、合并 1、持久化 1。所有线程池和队列有界，不按股票或事件无限创建线程。

long 使用低优先级独立展示分支，复用标准化、风险和 DeepSeek 原始五维结果，但不占用三策略评分 worker、不进入冻结写入。

事件包含 `event_id`、类型、主体键、交易日、阶段、策略、优先级、行情版本、配置版本、创建时间、截止时间、重试次数和载荷引用。幂等键为 `trade_date + phase + strategy + event_type + subject_key + data_version`；市场级主体使用固定值 `market`，个股级主体使用规范化六位代码。事件进入跨线程队列前必须取得并深层冻结 JSON 形态载荷，审计写出时再生成独立可序列化副本，调用方后续修改不得改变已入队或已预留事件。事件状态转换采用 compare-and-set，同一幂等键只能有一个有效执行者。

事件优先级：冻结 > 风险变化 > DeepSeek 补审 > 策略评分 > 候选行情 > 全市场行情 > 长期数据。普通行情队列满时合并同主体旧版本并保留最新版本；风险和冻结事件使用独立保留容量并先写入运行库，进程重启后重放，不能只依赖内存队列。队列容量、合并数、拒绝数和重放数必须进入状态指标。

生产组合根只创建一组数据采集 worker，东方财富、新浪、腾讯、Tushare 和 AKShare 五个逻辑来源 lane 共用该有界池。池内固定 6 个 worker：5 个普通来源 worker 与 5 个普通待处理槽位，另有 1 个仅供腾讯候选/TopK 定向报价使用的紧急 worker 与 1 个紧急待处理槽位；线程名前缀为 `source-data`，不得在运行主链中再叠加同类临时线程池。每个来源最多一个运行任务和一个按主体合并后的最新请求；配置声明的 worker 必须在流水线启动时成为真实消费者，各执行器同时限制运行任务和待处理任务，启动中途失败必须回收已启动线程。同一池内任务同步派生子任务时，仍有空闲 worker 才允许向共享队列展开；全部 worker 已占用时必须在当前 worker 内执行，禁止相互等待自身队列。long 的本地评分使用独立单 worker，并在 today/tomorrow/d25 的共享复核完成后再读取策略无关 DeepSeek 缓存，缓存未命中时才提交 long 补审。

事件落库状态固定按 `pending -> running -> success/failed/expired` compare-and-set；`expired` 只用于非冻结周期事件耗尽 deadline 的可观测降级。崩溃遗留的高优先级 `running` 事件先原子退回 `pending` 再重放，配置版本不匹配的旧事件必须失败关闭。同一事件只有 CAS 成功的执行者可进入业务链。退出时先关闭事件接收门并关闭内存队列，合并线程按优先级排空已接收事件和冻结写入，再关闭数据、标准化、策略、DeepSeek 与 long 执行器，持久化单写线程最后关闭；关闭完成后不得残留运行时 worker。

## 5. 交易时间线

- 09:15-09:30：共享预热，构建候选、历史因子和证据。
- 09:30-09:36：today 观察，不产生可执行动作。
- 09:36-10:30：today 主执行。
- 10:30-11:20：today 降级执行，提高阈值。
- 11:20：冻结 today 最终信号；此后不新增可执行推荐，只更新展示报价。
- 13:00-14:20：tomorrow、d25、long 预计算。
- 14:20-14:48：新入围、风险变化和证据变化补审。
- 14:48：停止提交生产 DeepSeek 请求。
- 14:49:50：强制刷新最终候选报价。
- 14:50：冻结 tomorrow、d25 及当日最终展示快照；long 只发布当前观察快照，不形成历史推荐。
- 15:00：保存收盘行情，只更新展示字段，不修改冻结策略。

所有业务时间使用 `Asia/Shanghai`。窗口统一采用左闭右开区间，单点任务以 `HH:MM:SS` 为计划时刻，文档中的“截止”表示该时刻起禁止新提交。调度延迟或进程重启错过冻结、DeepSeek 截止和收盘单点时，必须在同一交易日首次后续 tick 幂等补提交；14:49:50 最终候选报价只允许在 14:50 前补提交，冻结后不得以迟到报价重建候选。周期任务仍按第 6 节错过不补跑。交易日以 A 股交易日历判断，不能只用工作日代替；日历源失败时使用带版本和更新时间的本地缓存，缓存缺失或过期不得猜测交易日，只能进入降级只读状态。

## 7. 实时性、限流与降级

每条数据保存观察时间、来源时间、接收时间、生效时间、发布版本、载荷哈希、来源名称、字段缺失原因和终态；乱序、未来、无时区、空版本或截止后返回的结果不得覆盖新版本。实时字段与慢数据字段分域合并，慢数据不得覆盖更新的实时价格。

实时目标：

- TopK 价格：关键阶段 P95 年龄不超过 10 秒，其他交易阶段不超过 20 秒。
- 候选报价：主执行不超过 15 秒，其他阶段不超过 30 秒。
- 全市场快照：主执行不超过 60 秒，其他阶段不超过 120 秒。
- SSE 发布延迟 P95 不超过 2 秒。
- today 行情变化到评分发布 P95 不超过 15 秒。
- 11:20 today 锚点报价年龄不超过 20 秒，目标不超过 10 秒。
- 14:50 锚点报价年龄不超过 30 秒，目标不超过 10 秒。

数据年龄超过计划周期 2 倍标记 `stale`，超过 3 倍进入 `degraded`。today 行情超过 20 秒、tomorrow/d25 冻结行情超过 30 秒，只能观察。

全市场请求截止 20 秒、候选报价 3 秒、新闻事件 8 秒、Tushare SDK transport 8 秒、DeepSeek 默认 12 秒。同一来源和同一稳定缓存身份执行 single-flight；每个来源 lane 只有一个运行任务和一个 latest-wins 待处理任务，连续失败 3 次熔断 60 秒，半开时只允许一个探测请求。腾讯候选和 TopK lane 必须使用独立紧急执行位，历史、研究和全市场任务不得占用该执行位，避免普通 FIFO 排队先耗尽候选报价的 3 秒总截止。缓存统一使用有界 LRU、注入的 monotonic 时钟、stale-while-revalidate 和短期负缓存；负缓存只抑制重复刷新，必须保留并返回最近有效值、原始来源时间和数据版本，不能用失败条目覆盖有效值。缓存不得创建线程或持有响应对象、DataFrame、SDK 客户端和数据库连接。缓存身份固定包含数据集、来源、主体、非敏感请求指纹、交易日、规范阶段、来源契约版本、配置版本和 schema 版本，慢数据阶段统一为 `all_day`。

网络失败不得清空推荐。优先展示最近有效快照，并明确数据来源、年龄和降级原因。

## 10. 本地评分基础公式

定义 `B(x;l,o1,o2,u)`，参数必须满足 `l < o1 <= o2 < u`：`x <= l` 或 `x >= u` 得 0，`o1 <= x <= o2` 得 100，`l < x < o1` 时为 `100 * (x-l)/(o1-l)`，`o2 < x < u` 时为 `100 * (u-x)/(u-o2)`。无效参数使配置启动校验失败，缺失或非有限 `x` 走因子缺失规则。`P(x)` 为 0-100 横截面平均秩分，方向相反的因子必须在因子登记中显式声明并执行 `100-P(x)`。

```text
base_score = sum(component_score * component_weight)
local_score = clamp(base_score - local_risk_penalty, 0, 100)
```

组件和子组件权重都必须精确合计为 1.0，启动时校验；计算过程保留完整精度，统一舍入规则见第 17 节。

本地风险扣分上限 25：

- 近涨停拥挤 5。
- 量价背离 4。
- 高波动 3。

每个风险事实必须生成稳定 `risk_fact_id`，同一事实在本地规则中只能扣一次，必须展示风险代码、实际值、阈值、来源、证据时间和扣分。

`config/v2/strategy.json` 必须把上述区间规则展开为确定的风险表，每项登记 `risk_code`、适用策略、触发公式、阈值、严重度、扣分、最低置信度、证据有效期、互斥/叠加组和 `risk_fact_id` 生成字段。所有可变区间在配置中必须落到精确分段，不能由实现自行选择 3-8 或 5-15 内的数值；本地合计超过 25、DeepSeek 合计超过 30 时分别截断并保留截断前明细。

## 13. d25_plan 与 long_plan

d25 权重：动量 30%、趋势 25%、流动性 20%、执行 15%、不过热 10%。

- 动量：5/10/20 日强度 30%/30%/25% + 量价确认 15%。
- 趋势：MA20/60 结构 35% + 斜率 30% + 突破 20% + 行业趋势 15%。
- 执行：成交容量 40% + 振幅 30% + 价格可执行性 30%。
- 不过热：MA20 偏离反向 40% + 20 日涨幅不过热 35% + 低波动 25%。

20 日涨幅不高于 15% 时过热系数 1.0，15%-30% 线性降至 0.85，高于 30% 为 0.75。市场状态系数为风险偏好 1.03、中性 1.0、风险规避 0.92。d25 固定计算顺序为 `adjusted_base_score = clamp(base_score * overheat_factor * market_regime_factor, 0, 100)`，再执行 `local_score = clamp(adjusted_base_score - local_risk_penalty, 0, 100)`；市场状态分类公式必须登记在因子配置中，不允许运行时人工切换。

市场状态只使用同一 `data_version` 横截面中涨跌幅有限的股票，`market_breadth = 100 * 上涨股票数 / 有效股票数`；宽度不低于 60 为 `risk_on`，不高于 40 为 `risk_off`，其余为 `neutral`。没有有效样本时宽度保持 `null`、状态降级为中性并留存缺失，不得把缺失样本计作下跌。过热边界、线性端点、状态边界和三个状态系数全部写入 `strategy.json` 并参与策略版本哈希；`return_20d_not_overheated`、`d25_overheat_factor` 和 `market_regime_factor` 的登记必须与可执行公式一致，启动时拒绝矛盾配置。

long 使用受版本控制的固定名单，按价值 30%、成长 25%、质量 20%、行业/政策 15%、风险保护 10% 排序。五项本地输入按以下确定性公式生成：

- 财务只接受东方财富主要财务指标中公告时间不晚于观察时点、报告日距观察日不超过 550 天的最新记录。只有日期没有时分秒时按该日 23:59:59 处理，禁止在当日盘中提前使用。累计 EPS 和 ROE 按报告月 3/6/9/12 分别乘 4/2/4÷3/1 年化；其他月份记录无效。
- 价值先计算 `PE = 未复权实时价 / 年化基本 EPS` 和 `PB = 未复权实时价 / 每股净资产`。PE 不高于 10 得 100、达到 50 得 0，中间线性；PB 不高于 1 得 100、达到 8 得 0，中间线性。EPS、每股净资产或分母非正时对应子项缺失，价值为可用子项等权平均，两个子项都缺失时为 `null`。
- 成长使用营业总收入、归母净利润和扣非归母净利润同比百分点的可用值等权平均，`growth_score = clamp(50 + 2 * mean_growth, 0, 100)`；三项都缺失时为 `null`。
- 质量由 `clamp(50 + (年化 ROE - 10) * 2.5, 0, 100)` 与 `clamp(扣非归母净利润 / 归母净利润 * 100, 0, 100)` 的可用值等权平均；两项都缺失时为 `null`。营业收入同比不高于 -10、归母净利润同比不高于 -20 或扣非净利润同比不高于 -20 时 `financial_deterioration=1`；至少一项同比可用且均未触发时为 0，全部缺失时为 `null`。
- 个股公告只接受东方财富公告接口中带精确展示时间、发布时间不晚于观察时点且位于最近 180 天的最新 100 条。公告负面等级、减持等级以及政策正负关键词表必须写入配置，本地按命中的最高等级生成 0-3 严重度；无命中且接口成功为 0，接口失败保持 `null`。政策证据分固定为 `clamp(50 + 10 * (正向去重关键词命中数 - 负向去重关键词命中数), 0, 100)`，行业/政策分固定为行业强度 60% + 政策证据 40%，任一输入不可用时保持 `null`。
- 质押使用东方财富最新点时质押快照的累计质押占总股本百分点：低于 10 为 0，10-20 为 1，20-35 为 2，不低于 35 为 3。解禁使用观察日起未来 90 个自然日内各批次占总股本比例之和：低于 1% 为 0，1%-5% 为 1，5%-10% 为 2，不低于 10% 为 3。`reduction_or_unlock` 取减持公告与解禁等级最大值；任一来源已经给出正等级时可使用该等级，否则只有两个来源均成功才能确认 0，不能把来源失败写成无风险。
- 风险保护固定为低波动 50% + 低最大回撤 50%，任一项缺失时为 `null`。财务恶化、公告、质押、减持和解禁按第 9 节执行硬过滤，不进入本地风险扣分或风险保护，避免候选在剔除前被评分，也避免重复惩罚。DeepSeek 新发现且经证据规则验证的风险事实仍按第 15、17 节独立映射扣分或 veto。

完整财务研究只为 d25 和 long 候选按代码抓取；第 9 节所需公告、质押和解禁风险快照由四策略候选共享。每个来源使用 8 秒以内超时、有界 worker、每股最多一页/100 条公告和 10 分钟短期缓存；周期风险刷新只请求缺失或缓存已过期的代码，不得每轮强制重抓成功缓存并使固定排序后的后续候选长期饥饿。财务、公告聚合、质押和解禁的点时输入必须保存来源时间、接收时间、来源版本、脱敏摘要和派生值，风险相关公告另保留实际标题与发布时间；未来、过期、非有限或结构不合法的数据不得参与计算。单来源失败只使依赖该来源的值保持 `null` 并记录缺失/降级，不得清空最近有效行情或阻塞其他股票。所有 long 公式、窗口、阈值、关键词和缺失规则必须登记并参与策略版本，配置与执行不一致时启动失败。

目标价为可选人工配置，未配置显示 `--`，不得由模型虚构。long 只展示当前快照，不冻结、不产生执行动作、不写推荐历史、不参与日期查询；允许保存通用 DeepSeek 调用审计和短期缓存，但不得把它们解释为长期推荐记录。

## 14. DeepSeek 当前问题与改造目标

旧系统存在分析批次为零、调度租约长期停留 `running`、定性证据门控提前过滤全部候选、子进程状态不可见等问题。

新版取消调度子进程，调度线程直接提交有界 DeepSeek 任务。候选只要通过硬过滤且具有行情和至少一组结构化特征即可调用；新闻、公告不是调用资格。

缺少单类证据时仅将该维度设为 `unknown`，不能因此把整只股票提前 `abstain`。批次任务必须进入 `success`、`partial`、`failed`、`skipped` 或 `abandoned` 终态；每只股票另有 `applied`、`abstain`、`rejected` 或 `late` 结果，不能混用批次状态与候选结果。全部维度未知或加权置信覆盖不足时，候选结果为 `abstain` 并回退本地分。

生产复核采用受控双模型而不是开放式多智能体链：主审固定使用 `deepseek-v4-flash` 非思考模式，负责低延迟五维结构化输出；挑战者固定使用 `deepseek-v4-pro` 思考模式且 `reasoning_effort=high`，只复核第 15 节定义的关键候选。`deepseek-chat` 和 `deepseek-reasoner` 已停用，配置不得继续使用旧别名；迁移保持官方 `https://api.deepseek.com` base URL，只显式切换模型名和思考参数，依据为 [DeepSeek V4 官方迁移说明](https://api-docs.deepseek.com/news/news260424/)。主审发送 `temperature=0`，挑战者不得发送在思考模式下无效的 temperature/top_p 参数。主审不可用时回退当前有效本地结果；挑战者不可用时保留已经有效的主审结果，不得把候选或整版改成失败。

模型身份必须进入配置、缓存和审计：请求模型、模型角色 `primary/challenger`、思考模式、`reasoning_effort`、prompt/schema 版本任一变化都生成新缓存身份；响应实际模型与 `system_fingerprint` 作为缓存条目和冻结审计元数据保存，后续校准或版本比较不得跨实际身份混用。响应未返回的可选字段保持 `null`，不得用请求值冒充实际值。配置启动时拒绝旧模型别名、未知思考模式或主审与挑战者使用同一角色配置。

每只股票进入 prompt 的证据仍最多 16 条，首轮按固定槽位路由：结构化行情特征 1 条、尾盘分钟量价 1 条、财务与研究事实 3 条、监管/质押/解禁等风险事实 4 条、新闻与公告 7 条。空槽按“监管风险 -> 财务研究 -> 分钟量价 -> 新闻公告”顺序补给仍有合格证据的类别；补位后监管风险最多 6 条、财务研究最多 5 条、新闻公告最多 8 条，结构化行情和分钟量价仍各最多 1 条，总数不得超过 16。相同主体、事实类型、发布时间和内容摘要哈希的跨来源记录只保留来源质量更高且接收更早的一条。存在相反方向的有效材料时必须至少保留一条反证，不能只挑支持主审方向的证据。

证据只有在 `published_at <= observed_at` 且 `received_at <= observed_at`、来源版本存在、TTL 未过期时才可进入 prompt；任一时间缺失、未来到达、内容超限或 schema 不合法均记录缺失原因并排除。prompt 固定顺序为安全约束、schema、JSON 示例、稳定字段说明、候选数据；公共前缀逐字稳定，候选和证据按代码与证据 ID 排序，把变化数据放在末尾以利用 [DeepSeek 自动上下文缓存](https://api-docs.deepseek.com/guides/kv_cache)。调用审计保存官方 `usage.prompt_cache_hit_tokens` 与 `prompt_cache_miss_tokens`；缓存是尽力而为的成本和延迟优化，命中与否不得改变业务结果。[JSON Output](https://api-docs.deepseek.com/guides/json_mode/) 必须同时设置 `response_format=json_object`、明确要求 JSON 并提供最小示例；空内容、`finish_reason=length` 或截断 JSON 按非法 schema 处理，与现有修复共享最多两次物理尝试。

## 15. DeepSeek 五维规则

五维为价值质量、财务健康、资金量价、行业政策、风险质量。主审每维输出：

- `score`：0-100。
- `raw_confidence`：0-1，只表示模型自报把握，不得解释为真实正确概率；线上的兼容字段 `confidence` 在 schema 升级期必须与其数值相同。
- `flags`。
- `evidence_ids`。
- `assessment`。

每个维度的原始 `score` 和 `raw_confidence` 必须是有限数并分别位于 0-100、0-1；当前生产维度有效分仍为 `effective_i = 50 + (score_i - 50) * raw_confidence_i`，未知维度固定为 `score=50, raw_confidence=0, effective=50`，已知维度权重不因其他维度未知而重新归一化。预留的 `calibrated_confidence` 与 `calibration_version` 当前只允许为空或进入影子审计，不得参与生产融合；其训练、离线验证与晋级规则留待后续独立契约。

`rating`、`review_stage`、`challenger_status` 及模型自报置信度属于只读审计元数据，不参与动作、排序、扣分或 veto。生产动作只允许由规范化最终分、时间窗口、行情新鲜度、硬过滤和经本地规则验证的 veto 决定；TopK 同分顺序继续严格使用第 17 节固定键。

| 策略 | 价值质量 | 财务健康 | 资金量价 | 行业政策 | 风险质量 |
| --- | ---: | ---: | ---: | ---: | ---: |
| today | 10% | 10% | 40% | 15% | 25% |
| tomorrow | 15% | 20% | 25% | 20% | 20% |
| d25 | 20% | 25% | 20% | 20% | 15% |
| long | 30% | 30% | 10% | 20% | 10% |

对当前策略使用表中权重 `w_i`：

```text
deepseek_score = sum(effective_i * w_i)
confidence_coverage = sum(raw_confidence_i * w_i)
```

仅当 schema 完整、证据引用有效、结果在阶段截止前到达、候选非 `abstain`、至少两个维度已知且 `confidence_coverage >= 0.50` 时，`deepseek_applied=true`。阈值是策略契约的一部分；后续修改必须提升策略版本并通过离线点时检验。`partial` 批次中的单只股票只要独立满足上述条件仍可应用，其余股票逐只回退。

DeepSeek 不能新增股票。池外代码、无效证据引用和无法解析的 schema 结果直接拒绝。

风险维度除通用字段外，返回结构化 `risk_facts`，每项包含 `risk_code`、`severity`、`confidence`、`evidence_ids` 和 `assessment`。模型不得直接决定生产扣分；本地规则根据已验证事实映射 `deepseek_risk_penalty`，范围为 0-30。与本地风险集合具有相同 `risk_fact_id` 的事实不再进入 DeepSeek 数值扣分，避免重复处罚；连续风险质量分、离散风险扣分和 veto 必须分别展示。

高风险证据有效且置信度不低于 0.7 时才允许 `deepseek_veto=true`；模型输出不得直接决定 veto，本地风险表必须显式登记 veto 资格、允许证据类型和证据有效期后才能映射。缺数据、证据类型不匹配、证据过期、低置信度或单纯看空意见不得 veto。数值扣分同样必须有池内证据引用并达到对应风险规则的最低置信度，veto 可以在数值去重后继续作为独立安全门生效。

主审不得看到 `local_score`、本地排名、动作阈值、其他候选结果或目标价，避免锚定和顺从本地结论。挑战者只处理排名保护集合、接近 TopK、主审与本地方向冲突、证据内部冲突或出现新高风险事实的候选；它可以看到主审每维的方向标签、事实主张和证据引用，但不得看到主审数值分、主审置信度、最终分或排名。挑战者逐维只输出 `verdict=confirm/contradict/insufficient`、`raw_confidence`、`evidence_ids` 和短原因码，不重新生成股票分数、目标价、扣分、veto 或交易动作。

两阶段合并固定为：`confirm` 保留主审分且该维置信度取主审与挑战者 `raw_confidence` 的较小值；具有有效反证引用的 `contradict` 把该维设为 `unknown`，不得反向创造新分；`insufficient`、挑战者失败、超时、预算耗尽或迟到均保留主审结果并记录挑战者缺失。挑战者不能新增候选，不能改变已经由本地规则验证的风险事实，也不能绕过 `hybrid/local_degraded` 整版一致性门。

## 17. 合并、动作与 TopK

融合公式版本固定为 `fusion_v2_local68_deepseek32`。`local_score` 已按第 10 节扣除本地风险，最终融合不得再次扣除 `local_risk_penalty`。候选级 `fusion_applied = deepseek_applied and fusion_mode == "hybrid"`：

```text
if fusion_applied:
    raw_final_score = local_score * 0.68 + deepseek_score * 0.32 - deepseek_risk_penalty
else:
    raw_final_score = local_score

final_score = round_half_up(clamp(raw_final_score, 0, 100), 2)
```

`deepseek_risk_penalty` 是融合后的绝对分值，不乘以 0.32。DeepSeek 失败、弃权、超时、额度耗尽、迟到、覆盖不足或 schema 无效时，该候选 `final_score = round_half_up(local_score, 2)`、DeepSeek 扣分为 0、普通 DeepSeek veto 不生效；已经由证据规则验证的重大安全 veto 仍可独立阻止执行。排名、动作判断、持久化和 API 都使用同一个两位小数规范分，不允许界面舍入值与门槛判断值不同。

每个 `strategy + data_version` 只能发布一种 `fusion_mode`：

- `hybrid`：排名保护集合中的候选均已得到 `applied` 或 `abstain` 终态，`applied` 使用 68/32 公式，`abstain` 使用本地回退。
- `local_degraded`：保护集合中仍有失败、迟到、拒绝或未完成候选，整版排名统一使用本地分，避免未复核股票因回退而系统性占优；页面明确显示降级原因。

排名保护集合为本地分前 18 名与距当前策略动作门槛 5 分以内候选的并集。模式切换必须产生新发布版本，冻结时模式一并固化。

默认 TopK 为 10，接口允许 0-18；`top_n=0` 合法并返回空列表。排序固定为最终分降序、本地分降序、股票代码升序；单一行业最多 3 只，不足时不得降低门槛补满。

- today 主窗口：`final_score >= 70` 可执行。
- today 降级窗口：`final_score >= 76` 可执行。
- tomorrow：`final_score >= 72` 可执行。
- d25：`final_score >= 70` 可执行。
- 距门槛 5 分以内：观察。
- 其他情况：不可执行。

所有可执行项必须满足硬过滤、时间窗口、行情新鲜、无 veto 和周期匹配。

上述动作门槛是 v14 当前候选值。切换生产前必须使用冻结点时输入完成离线重放，并至少报告各策略分数分布、推荐数、空推荐比例、TopK 变化、DeepSeek 覆盖、`local_degraded` 比例和风险拦截率；门槛未通过预注册验收时必须在本文和 `strategy_version` 中同步修订，禁止运行时自动调参或只改配置不升版本。`trader-cli threshold-report --snapshot <absolute-path>` 可重复传入同一策略版本和融合版本的冻结快照，报告必须基于完整回放候选而非仅基于已入选 TopK；混合版本输入必须拒绝，避免把不同规则的数据合并为伪基线。

`strategy`、`recommend_date`、`stock_code`、`rank`、`anchor_price`、`anchor_daily_return_pct`、`snapshot_id`

- `data_source_health`：来源成功率、延迟、熔断和数据年龄。

非冻结周期事件在执行前或执行中耗尽 deadline 时必须进入独立 `expired` 终态并计数，不得写入全局最近错误；已完成的旧快照继续可读。deadline 后返回的全市场特征不得提交到候选池或特征缓存。真实来源、持久化或业务计算失败仍进入 `failed` 并记录最近错误。冻结事件不使用该过期降级路径，继续以 11:20/14:50 边界和迟到写保护为准。

SQLite 与 JSON 只允许单写线程操作。实时草稿写入可替换的 `published/`；冻结时先在 SQLite 事务中以预计算路径和 SHA-256 创建 `staged` manifest，再写同目录临时文件、刷新并原子改名，最后在第二个 SQLite 事务中写推荐行、已发布指针并把 manifest 改为 `committed`。启动恢复扫描 `staged` manifest、孤儿文件、哈希不符和缺失文件：完整项幂等提交，不完整项隔离并回退最近有效快照，禁止静默覆盖。

## 19. Web API

公开接口：

- `GET /api/status`
- `GET /api/recommendations/<strategy>?date=YYYY-MM-DD&top_n=10`
- `GET /api/recommendation-dates?strategy=<strategy>`
- `GET /api/events?cursor=<cursor>&limit=100`
- `GET /api/events/stream`

`strategy` 只接受 `today`、`tomorrow`、`d25`、`long`；long 只允许当前日期或省略日期，不出现在历史日期接口。`top_n` 必须为 0-18 的整数，日期、游标或策略无效返回结构化 400，快照不存在返回 404，服务已启动但尚无快照返回 200 空结果并带 `not_ready` 状态。所有响应使用版本化 envelope，至少包含 `schema_version`、`snapshot_id`、`published_at`、`data_version`、`strategy_version`、`fusion_version`、`fusion_mode`、`stale`、`degraded_reasons`、`items` 和 `error`。推荐 envelope 还必须返回 `requested_date`、`current_trade_date` 和 `historical`；当前请求降级到上一交易日时同时返回 `fallback_date` 与 `fallback_reason`，400/404 错误保留可解析的策略和请求日期上下文。

候选项的 DeepSeek 审计字段增加 `review_stage`、`challenger_status`、`evidence_manifest_hash`、`requested_model`、`actual_model`、`thinking_mode`、`raw_confidence`、可空的 `calibrated_confidence` 与 `calibration_version`。旧冻结缺少这些字段时按 `null/not_run` 兼容读取；API 不返回思维链正文、完整外部载荷或密钥。挑战者失败只显示候选级状态和原因，不得把有效主审伪装为“未复核”。

v15 在现有 envelope 的 `metadata` 中以加法返回 `merge_epoch`、`source_versions`、逐股票 `field_sources`、`market_conflicts`、`market_missing_reasons`、`market_degraded_reasons`、`market_observed_at` 和 `tushare_reference_versions`。候选项以加法返回 `cross_source_deviation_pct`、`cross_source_verified`、`board`、`board_source`、`board_reliability`、`exchange`、`listing_date`、`listing_age_sessions`、重上市/退市整理首日标记、`has_price_limit`、`exchange_limit_pct`、`strategy_hot_cap_pct`、`rule_version`、`rule_effective_date` 和 `execution_restrictions`。旧冻结缺失时保持 `null`、空值或旧算法默认，不得读取时伪造为已验证主数据。

v15 新冻结的回放算法标识固定为 `engine_v15_parallel_market_data_2026_07`。既有 `engine_v10_section9_hard_filter_2026_07` 冻结继续使用 v14 过滤兼容路径，保留原成长板过滤码和原偏差硬过滤语义；不得用 v15 的板块过滤码、上市日龄门或只能观察规则重解释旧冻结。冻结外层追加的 `freeze_anchor` 与 v15 多源审计字段不参与评分算法投影，但必须继续受冻结 JSON 和 manifest SHA-256 完整性校验。

Web 处理器只读取已发布快照，不允许直接抓取行情、计算评分或调用 DeepSeek。当前快照支持 ETag；相同版本返回 304，避免轮询重复传输。ETag 身份必须包含当前交易日、快照、overlay 和 fallback 信息，跨交易日不得继续复用上一日缓存身份。前端在 SSE 正常时不得重复轮询完整推荐响应，但必须使用既有状态心跳中的策略快照身份对账；发现当前页面仍是旧快照或上一交易日 fallback 且服务端已发布新身份时，立即补拉一次当前推荐。历史响应的 `trade_date` 必须与 `requested_date` 完全一致；历史列表可用独立当前行情展示“今日涨跌”和“锚点至今”，但不得把该行情伪装成历史快照的 `live_overlay` 或改写冻结记录。

`/api/events` 是有界审计查询，`/api/events/stream` 是 SSE。SSE 只推送已发布版本，事件 `id` 使用单调发布序号，支持 `Last-Event-ID`；游标仍在保留窗口内时补发，过旧时发送 `resync_required` 并由前端获取完整快照。心跳 15 秒；断线后前端每 15 秒轮询，恢复 SSE 后停止轮询，避免双重刷新。服务端必须限制单客户端缓冲和总连接数，慢客户端不得阻塞发布线程。

## 24. 分阶段实施

### 24.0 “继续”命令的整节交付协议

用户每发送一次“继续”或语义等价的继续执行指令，都推进计划或需求文档中下一个完整未完成章节，以同级标题为章节边界，并完成章节内全部明确子项。同一章节的多个可独立验收子项必须作为一个整节任务统一实现、回归、Review、提交和推送，不得完成首个子项后提前停止，也不得顺带处理相邻章节或无关优化。直接下达的新任务同样形成独立交付批次；用户明确指定完整章节时按整节交付，否则多个互不相关目标仍拆成多个任务逐次交付。

每个整节任务固定执行以下闭环：

```text
记录已推送基线和已有工作树变更
  -> 选择下一完整未完成章节并限定文件范围
  -> 完成章节内全部明确子项与针对性验证
  -> 独立 Review
  -> 修复发现并重新 Review，直到零已知问题
  -> 更新 CHANGELOG.md
  -> 创建一个 Conventional Commit
  -> 推送当前跟踪分支并核对本地/上游提交一致
  -> 停止，等待下一次“继续”
```

本章节任务 Review 必须以上一次已推送提交为基线检查完整 diff，不能由阶段末或整体计划 Review 代替。Review 通过只结束审查计划项，章节内全部子项完成前交付任务仍保持进行中。每个整节任务只允许一个新提交和一次对应的成功推送记录，不得把多个“继续”批次或相邻章节合并、不得修改已推送任务历史。网络、权限或远端拒绝导致推送失败时，保留本地提交并明确报告阻塞；推送重试仍属于当前章节任务，直到本地 `HEAD` 与 `@{upstream}` 哈希一致后任务才算闭合，才能开始下一章节。

每个独立交付批次还必须在 `CHANGELOG.md` 的 `Unreleased` 中归纳用户提出的问题、原因或现状判断、修改说明、验证证据和剩余风险/后续项，不能只罗列文件名或提交信息。相同根因的连续反馈可以合并，未确认的判断必须明确标为待验证；业务、公式、API 或验收契约发生变化时仍须同步本文件。归纳内容不得包含密钥、完整外部响应或其他敏感数据。

### 26.1 v15 来源职责与并发生命周期

五个来源固定为独立逻辑 lane：

| 来源 | lane | 唯一职责 | 触发方式 |
| --- | --- | --- | --- |
| 东方财富 | `eastmoney-source` | 全市场、分钟和历史基础行情 | 第 6 节行情周期 |
| 新浪 | `sina-source` | 全市场同时校验和字段回退 | 与东方财富同时触发 |
| 腾讯 | `tencent-source` | 候选和 TopK 定向报价 | 仅候选/TopK 周期 |
| Tushare | `tushare-source` | 证券主数据、交易日历、前复权日线、日度估值和财务指标 | 盘前、盘后或慢数据 TTL |
| AKShare | `akshare-source` | 行业、新闻、公告和研究数据 | 候选范围或慢数据 TTL |

组合根只实例化一个线程前缀为 `source-data` 的有界执行器，包含 5 个普通来源 worker 与 5 个普通待处理槽位，以及 1 个腾讯候选/TopK 紧急 worker 与 1 个紧急待处理槽位，所有 lane 共享其生命周期。每个 lane 最多一个运行任务和一个合并后的最新请求；队列满或同源仍在运行时按主体只保留最新 `observed_at`，不补跑旧周期。启动、停止和等待由运行时显式控制，`create_app()` 不启动 worker、网络、数据库或文件写入。单个永久阻塞普通来源最多占用一个普通 worker，不得阻塞其他普通来源或紧急报价；腾讯、Tushare 或 AKShare 失败不得阻止东方财富/新浪生成统一快照。

Tushare 是独立可选 extra，官方 SDK transport timeout 固定为 8 秒。Token 先读 `TUSHARE_TOKEN`，其次读 `TUSHARE_TOKEN_FILE` 指向的单行普通文件；POSIX 下拒绝 group/other 可读文件。未安装、无 Token、429、额度错误或超时统一记录脱敏降级，不写入配置、日志、SQLite、冻结或 API，也不触发 Tushare 高频报价。

### 26.2 v15 单源观测、缓存与确定性合并

所有适配器先输出不可变 `SourceObservation`，字段至少包含来源、主体、观察/来源/接收/生效时间、响应数据版本、不可变字段映射、缺失原因、载荷哈希、终态和脱敏错误码。datetime 必须带时区并按 `Asia/Shanghai` 业务语义比较；跨线程前复制成不可变 JSON 形态，未来、无时区、空版本、非法代码、非有限核心值和 deadline 后完成的观测不得进入缓存、统一快照或冻结。

应用层定义唯一缓存身份 builder，身份固定为：

```text
dataset + source + subject_key + request_fingerprint + trade_date + phase
+ source_contract_version + config_version + schema_version
```

请求指纹使用排序后的代码、字段、复权、窗口和其他非敏感参数的规范 JSON SHA-256；主数据、日历、历史、估值和财务阶段固定为 `all_day`。响应后才可获得的 `data_version` 只保存于条目和值身份，不得预猜为请求键。统一缓存实现必须是注入 monotonic 时钟的有界 LRU，支持 fresh、stale-while-revalidate、degraded、短期负缓存、条目上限、缓存组字节上限和临时有界 single-flight 注册表；注册表完成、失败、超时、取消或停止后都必须清空。估算字节固定为缓存身份规范 JSON 与不可变载荷规范 JSON 的 UTF-8 长度之和，插入时只计算一次。

`runtime.json.market_data.cache_policy` 是 TTL、动作年龄、负缓存、条目容量、组字节容量和淘汰策略的唯一真相源；`runtime.json.performance_budgets` 是固定负载和延迟预算的唯一真相源。loader 拒绝缺失键、未知键、非正容量、动作年龄小于 TTL、缓存组之和不等于总量、数据集集合漂移、固定工作负载或轮次漂移。14:49:50 绕过 fresh 行情缓存，但继续复用 single-flight 并执行 14:50 deadline CAS；缓存过期或降级不得改写原始来源时间、风险事实、排名和冻结身份。

纯合并函数按股票一次索引各来源，目标复杂度为 O(S*N)，规则固定为：

1. 先排除非法或迟到观测，再按 `source_time`、`received_at` 选择较新字段；跨来源同时点按 `source_priority` 选择，`data_version` 只作为同来源同时点的稳定末级比较，禁止比较不同供应商无共同语义的版本字符串来改变来源优先级；以上键全部相同时仅用 `payload_hash` 消除输入顺序差异，不把哈希解释为业务新旧。
2. 同时点全市场实时字段优先东方财富，其次新浪；新鲜候选/TopK 实时字段优先腾讯。
3. Tushare 与 AKShare 慢数据只能补充其职责字段，不得覆盖更晚的实时价格。
4. 来源间价格偏差不超过 0.50% 可通过；大于 0.50% 时，腾讯定向报价必须与东方财富或新浪至少一个全市场观测的偏差不超过 0.50% 才算复核成功，否则记录冲突并只能观察，禁止用腾讯选中价与自身比较完成自证。
5. 相同有效输入不论完成顺序都产生相同逐字段来源、来源版本、冲突、缺失、`merge_epoch` 和规范 JSON 哈希。
6. 所有来源失败时返回最近有效 `CanonicalMarketSnapshot` 并增加降级原因，不返回空集合；迟到结果只可进入脱敏审计。

缓存固定数据集和 schema v5 数值为：

| 数据集 | 刷新 TTL | 动作/降级边界 | 负缓存 TTL | 条目容量 | 缓存组 | 持久恢复 |
| --- | ---: | ---: | ---: | ---: | --- | ---: |
| `full_market_quotes` | 当前 `full_market` cadence | cadence 3 倍 | 10 秒 | 每源 6000 | `realtime_observations` | 否 |
| `candidate_quotes` | 当前 `candidate_quotes` cadence | cadence 3 倍 | 3 秒 | 每源 360 | `realtime_observations` | 否 |
| `intraday_minutes` | 45 秒 | 90 秒 | 45 秒 | 每源 360 | `history_minutes_research` | 否 |
| `research_success` | 600 秒 | 1200 秒 | 60 秒 | 每源 360 | `history_minutes_research` | 既有证据文件 |
| `research_failure` | 60 秒 | 60 秒 | 60 秒 | 每源 360 | `history_minutes_research` | 否 |
| `daily_history` | 21600 秒 | 86400 秒 | 60 秒 | 每源 360 | `history_minutes_research` | 否 |
| `security_master_calendar` | 86400 秒 | 86400 秒 | 300 秒 | 每源 6000 | `history_minutes_research` | 否 |
| `daily_valuation_financials` | 86400 秒 | 86400 秒 | 300 秒 | 每源 360 | `history_minutes_research` | 否 |

缓存组固定为实时观测 64 MiB、历史/分钟/研究 128 MiB、为 v16 预留但 v15 不写入的板块候选评分 64 MiB，总量 256 MiB。条目数和所属组字节数任一先到上限即淘汰。v15 只有现有研究证据文件用于跨进程恢复；历史日线、Tushare 主数据/日历和估值财务不伪装成已持久化命中，进程重启后按来源 lane 和统一缓存策略重新加载。持久文件不得另行决定生产 TTL、容量或覆盖内存 LRU。实时行情 stale 和 degraded 精确边界为 cadence 的 2 倍和 3 倍，精确相等仍处于较轻状态。超出动作边界不得生成新推荐，只读 Web 仍展示最近有效已发布快照和原始时间。

Tushare 主数据和日历按上述 86400 秒策略缓存；交易日历按最多 3650 个自然日一个请求、最多 8 个请求分段，避免 SDK 单次行数截断且保持物理调用有界。前复权日线、日度估值和财务指标只按候选代码在盘前、盘后或慢 TTL 触发，同一 Tushare lane 内执行；只有日期没有时分秒的日线、估值、公告/财务记录一律在 `Asia/Shanghai` 当日 23:59:59 生效，盘中不得提前使用。v15 可把这些字段作为结构化审计和最近有效历史回退输入，但不得改变 v14 候选、评分、动作门槛或 TopK。

rule_effective_date

### 26.4 板内总体与点时纪律

v16 横截面总体固定为：

```text
trade_date + phase + board + data_version
```

流动性、换手、波动、回撤、趋势、相对强度、同行差、领先滞后和行业强度均在板内截尾、归一化和排序。主板、创业板和科创板不得共享分位点、换手阈值或波动总体；某板样本不足时不得借用另一板结果。

实时价格、限制价和冻结锚点使用未复权价格；历史收益、均线、波动和回撤使用观察时点可获得的前复权序列；行业、证券状态、财务和规则按实际生效时间进入特征。每次输入必须绑定数据、规则、行业、配置和因子 schema 版本。缺失因子仍按中性 50 进入候选计算，但原始值保持 `null`，记录 `missing_fields` 和 `missing_reasons`，不得把中性占位伪装成真实观测。

### 26.5 同行、领先滞后和换手因子

同行收益差固定为：

```text
peer_return(board, industry, horizon)
    = median(return_j for j in valid_peers and j != stock)

peer_gap(stock, horizon)
    = stock_return - peer_return
```

同行必须位于同一板块和观察时点有效行业，排除股票自身；行业有效股票不足 10 只时保持缺失。today 使用 1/3/5 日，tomorrow 使用 5/20 日，d25 使用 20/60 日。负差补涨和正差回落是互斥方向，不得同时加分或看完结果再选择方向。

同行领先滞后固定为：

```text
leader_group
    = 同板同行业按过去 20 日成交额中位数排序的前 20%

leader_gap
    = leader_group_return - stock_return
```

行业至少 10 只有效股票、领先组至少 3 只，目标股票必须从领先组收益计算中排除。与行业趋势重复且没有独立信息时不得重复加分，必须保留缺失或去重审计。

换手和成交冲击固定为：

```text
turnover_shock_20 = current_turnover / median(turnover, 20)
amount_shock_20 = current_amount / median(amount, 20)
volume_price_confirmation = sign(return) * normalized_amount_shock
```

分母缺失、非正或非有限时保持缺失。高换手延续和极高换手反转属于互斥方向。成交量、成交额和换手相关指标统一命名为 `flow_proxy`，不得解释为主力、机构或散户资金。DeepSeek 只能解释这些结构化值，不得生成或改写因子。

本节只授权生成、冻结和审计上述结构化特征，不授权任何互斥方向进入当前生产评分；同行补涨/回落、高换手延续/极端换手反转的生产方向必须在后续独立业务契约中选定并提升策略版本，不能由实现者、配置或 DeepSeek 自行决定。
