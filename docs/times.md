# 实时行情与推荐链路性能优化计划

状态：非权威执行计划，尚未实施。

本文记录从 SDK/API 获取股票信息，到行情统一、候选/评分、P6/SSE 发布和浏览器绘制的
性能审查结论与后续实施顺序。产品、架构、时间线、API、运维和验收仍以
[软件业务设计文档](software-business-design.md) 为唯一权威；候选、过滤、评分、风险、
融合和排名仍以[荐股策略文档](recommendation-strategy.md) 为唯一权威。本文不得覆盖两份
权威文档；实施时如改变公共接口、运行契约或策略行为，必须先同步修改相应权威文档。

## 1. 目标与范围

优化目标固定为“行情和本地推荐优先实时”：

- TopK 行情在供应商能够稳定返回时按现有 1 秒计划刷新；
- 本地评分完成后先形成可见草稿，不等待 DeepSeek 才首次展示；
- DeepSeek 作为异步升级，返回后增量发布融合结果；
- 冻结时间、冻结后不可覆盖、188 次物理请求预算、固定融合公式和本地风险规则不变；
- Web 请求继续只读，不得在 HTTP 请求内抓行情、评分、调用 DeepSeek 或写数据库。

本计划不包含移动端、多租户、真实下单、新策略、自动调参或收益承诺。

## 2. 当前证据与瓶颈结论

### 2.1 暂不优先优化的部分

现有 P6、Web API 和 SSE 已有明显余量：

| 指标 | 当前 P95 | 现有门限 | 判断 |
| --- | ---: | ---: | --- |
| P6 到 SSE 入队 | 4.357ms | 100ms | 非主要瓶颈 |
| 当前推荐 API | 2.382ms | 200ms | 非主要瓶颈 |
| 单次价格增量相对完整 18 行响应 | 减少 89.655% | 越小越好 | 保持增量协议 |

在单用户本地 PC 范围内，不应优先投入复杂的多节点缓存、消息中间件、SSE 压缩或大规模
订阅优化。

### 2.2 P0：定向报价重复重建全市场

腾讯候选或 TopK 报价只更新 18 至 360 只股票，但当前提交路径仍可能：

1. 将定向结果 overlay 到完整 canonical snapshot；
2. 重建约 5500 行列式批次；
3. 比较完整 change set；
4. 重建全量代码索引和规范哈希。

专业固定负载的 P95 为：

| 阶段 | 5500 行 P95 |
| --- | ---: |
| 行情标准化 | 169.247ms |
| 两来源融合 | 675.536ms |
| canonical snapshot | 1219.953ms |

完整 canonical 路径已经接近 1 秒 TopK cadence，定向报价若复用该路径，会让供应商请求之外
的本地处理成为实时展示瓶颈。

### 2.3 P0：DeepSeek 阻塞本地首次发布

当前评分编排会并行提交各策略 DeepSeek future，但随后同步等待结果，最后才统一生成并发布
快照。影响包括：

- 已完成的本地评分不能立即展示；
- 外部模型的 12 秒 deadline 可能直接进入首屏延迟；
- 单一 merge/event 线程等待期间，TopK 报价事件可能排队；
- 各策略按提交顺序读取 future，存在已完成策略被较慢策略阻塞的队头等待。

这与“DeepSeek 失败不能阻塞本地推荐”的运行目标不一致，应作为首要编排优化。

### 2.4 P0：SDK 连接与分页调度

- 东方财富、新浪和腾讯适配器通常每个请求创建并关闭一个 `requests.Session`，无法持续复用
  TCP/TLS 连接。
- 东方财富全市场任务运行在它所借用的数据池线程时，会为避免嵌套线程池死锁而串行抓取
  剩余页面。
- 新浪全市场分页会临时创建自己的线程池，实际线程数、生命周期和与腾讯紧急报价的资源
  竞争没有统一控制。
- 当前来源延迟指标能看到供应商总耗时，但不能拆分排队、连接、下载、解析和本地提交。

### 2.5 P1：浏览器增量消息触发整表重绘

`overlay_patch` 只携带少量报价变化，但当前实时页面仍会重新生成表头、推荐表和观察表的
完整 HTML。18 行不至于造成白屏，但每秒重绘会增加 DOM 分配、布局、焦点丢失和低性能
桌面浏览器抖动。现有浏览器诊断只统计请求、字节和 resync，没有记录消息接收到下一帧
绘制的耗时。

### 2.6 P1：正式性能门禁没有运行真实链路

正式 `perf-check` 中部分操作是简单 Polars 构造、self-join 或 JSON 序列化；
`board_ready_to_draft`、`quote_to_draft` 和 `deepseek_to_hybrid` 目前没有实际执行生产
provider、融合、评分、P6、SSE 和浏览器路径。小型合成结果与 5500 行专业 runner 的差异
较大，因此不能只依据合成指标判断实时链路没有回退。

## 3. T1：真实延迟瀑布与性能门禁

### 3.1 运行观测

为每个行情/评分周期生成有界关联身份，并记录：

1. 计划与进入来源队列；
2. SDK 请求开始与响应接收；
3. 标准化、融合和 canonical 提交；
4. 候选准备、本地评分和 DeepSeek 提交/完成；
5. P6 接纳、SSE 入队；
6. 浏览器收到 patch 和下一次 `requestAnimationFrame`。

`/api/status` 只增加有界的样本数、P50、P95、max、超时、被替代和丢弃计数，不保存完整
请求/响应、密钥或个股敏感运行载荷。外部供应商耗时和本地处理耗时必须分开报告。

### 3.2 真实门禁

用生产函数和固定离线输入替换合成占位操作，覆盖：

- 5500 行东方财富/新浪全市场；
- 360 行候选和 120 行候选报价；
- 18 行 TopK overlay；
- 三策略本地评分；
- DeepSeek 固定延迟/失败替身；
- P6、SSE 编码和真实浏览器 DOM patch。

新增或收紧以下 P95 门禁：

| 指标 | 目标 |
| --- | ---: |
| 5500 行标准化 | ≤250ms |
| 两来源融合 | ≤600ms |
| canonical snapshot | ≤900ms |
| 360 行 targeted overlay commit | ≤100ms |
| board-ready 到本地草稿 | ≤500ms |
| 报价到本地草稿 | ≤5s |
| DeepSeek 完成到 hybrid | ≤1s |
| SSE 接收到浏览器绘制 | ≤100ms |
| TopK 关键阶段来源年龄 | ≤2s |

100 tick 分配增长继续为 0%，逻辑缓存和进程 RSS 继续受现有 248 MiB/384 MiB 契约约束。

## 4. T2：SDK 连接复用与有界分页

### 4.1 六线程实时数据预算

保持实时数据线程总数为 6，不通过增加隐藏线程换取表面时延：

- 腾讯紧急候选/TopK 报价池：1；
- 东方财富/新浪来源协调池：2；
- 东方财富/新浪共享分页池：3。

`SourceLaneRegistry` 按来源绑定执行器。腾讯紧急报价不与全市场分页共用可被全部占满的
协调线程；参考/低频任务使用普通优先级，不得抢占腾讯紧急池。

### 4.2 连接生命周期

- 每个供应商使用按工作线程隔离的长生命周期 Session，复用 TCP/TLS；
- 禁止多个并发线程共享同一个 `requests.Session`；
- 保持请求级 timeout、deadline、熔断、负缓存、latest-wins 和既有重试上限；
- 先停止 source lane 和 worker，再统一关闭全部 Session；
- `create_app()` 仍不创建 Session、线程或网络连接，所有资源只由组合根显式装配。

### 4.3 分页行为

- 东方财富和新浪使用同一有界分页池，不再临时创建额外执行器；
- 东方财富完整实时页使用较高任务优先级，新浪保持回退和交叉验证能力；
- 任一来源只有完整、同代、deadline 内的批次才能原子发布；
- 某页失败时不得发布拼接残缺的全市场快照，继续使用最近有效值并标记降级。

## 5. T3：列式全市场与定向 dirty 提交

### 5.1 全市场路径

- 完整供应商批次从原始行直接进入严格类型列式标准化；
- 避免为每行反复执行 `Polars -> dict/SourceObservation/hash -> Polars`；
- 以列式表达式完成可向量化的来源选择、偏差和基础规则；
- 只在进入应用层候选或构造最终 canonical 领域视图时物化 `MarketQuote`；
- partial、reference、Tencent 和 degraded 输入继续保留等价 scalar 回退。

### 5.2 定向报价路径

腾讯候选/TopK 报价只处理 dirty codes：

- 按代码融合定向观测与现有来源观测；
- 原地替换代码索引中的对应不可变报价；
- 列式批次仅删除并补入 dirty rows；
- 维护每行规范哈希，并从排序后的行哈希生成确定性内容身份；
- 只扩张受影响的板块、行业、风险和字段族；
- 不重复规范序列化全部 5500 行字段来源。

优化前后必须证明相同有效输入产生相同报价、字段来源、冲突、缺失原因、merge epoch 和规范
业务哈希。

## 6. T4：本地推荐先发布、DeepSeek 异步升级

### 6.1 发布状态

公共融合状态拟增加 `local_pending`：

- `local_pending`：本地评分、风险和下行保护已完成，DeepSeek 尚在处理；
- `hybrid`：合格 DeepSeek 结构化事实已由本地规则融合；
- `local_degraded`：DeepSeek 失败、超时、截止或没有达到应用条件。

本地 pending 结果仍执行全部本地下行保护；页面明确标记“模型复核中”，不得伪装成已完成
hybrid。该枚举扩展在实施前必须同步权威文档、API 契约和兼容测试。

### 6.2 异步完成事件

评分事件完成本地 `PreparedSnapshot` 后立即发布 `local_pending`，随后提交 DeepSeek：

1. DeepSeek worker 完成回调只写入有界 completion registry；
2. 回调向主事件队列提交高优先级 `deepseek_result`；
3. 单一 merge/event 线程验证策略、交易日、输入版本、配置版本、deadline 和冻结状态；
4. 验证通过后发布新的 hybrid 或 terminal local-degraded snapshot；
5. 每个策略最多保留一个最新周期，旧周期完成结果直接丢弃并计数。

不同发布阶段使用不同 `snapshot_id`，通过现有 base snapshot 增量协议升级。新 snapshot 发布
后立即用运行态最新报价重建对应 overlay，避免短暂回退到评分锚点价格。

### 6.3 截止、失败与冻结

- DeepSeek 失败不撤回已经发布的本地结果；
- DeepSeek cutoff 将仍 pending 的周期终态化为 `local_degraded`；
- 冻结前先终态化对应 pending 周期，再执行正式冻结；
- 冻结后到达的 completion 只能记为迟到丢弃，不得修改 P6、正式记录、checkpoint 或 SSE；
- 重启不恢复无法序列化的内存 completion，已有本地/正式快照继续可读并显式降级。

## 7. T5：浏览器逐行、逐帧更新

- 在报价单元格增加稳定字段身份，只更新价格、涨跌幅、锚点收益、来源和时间；
- 推荐 upsert/removal 只插入、替换、删除和按 rank 重排涉及的行；
- 表头只在首次加载、历史/当前切换或 schema 变化时重建；
- 同一动画帧内的多个 overlay 合并，每只股票只保留最新版本；
- 更新时保持键盘焦点、选中行和详情抽屉状态；
- resync、策略切换、历史切换和无法验证的 patch 继续执行完整 GET/完整渲染。

浏览器诊断增加 patch-to-paint、变更行数、完整重绘次数和超过 50ms 的 long task 计数，
并在 1280x720、1440x900、1920x1080 三档桌面视口验收。

## 8. 回归与验收矩阵

### 8.1 SDK 与资源

- 连接复用、线程隔离、关闭顺序、分页并发上限；
- 腾讯紧急报价不被两路全市场请求阻塞；
- timeout、deadline、取消、熔断、latest-wins 和无隐藏重试；
- 停止后没有存活的非 daemon worker 或未关闭 Session。

### 8.2 行情与确定性

- 5500 行双来源完整路径；
- 360/120/18 行定向 dirty 更新；
- 乱序完成、重复代码、来源部分失败、异常/未来时间；
- scalar/columnar 等价、规范哈希和 epoch 一致；
- 100 tick 内存、缓存和对象增长保持有界。

### 8.3 推荐与冻结

- DeepSeek 延迟 12 秒时本地草稿仍按门限发布，TopK 事件不中断；
- 多策略 future 乱序完成时分别升级，不发生队头等待；
- 成功、abstain、schema 失败、timeout、预算耗尽和 cutoff；
- 新评分周期替代旧周期、迟到 completion 丢弃；
- 11:20 和 14:50 冻结前终态化、冻结后不可覆盖；
- 固定融合向量继续为 `83.40`，重试和失败继续计入 188 次物理预算。

### 8.4 Web 与交付

- SSE 游标恢复、慢客户端、base mismatch、ETag resync；
- overlay 每秒更新不触发整表重绘或额外完整推荐 GET；
- 三档桌面无白屏、重叠、页面横向溢出和明显布局跳动；
- `make format-check`、`make lint`、`make type-check`、`make test`、`make package`；
- 仓库外安装 wheel 后导入、CLI、配置、模板、CSS、JavaScript、SVG 和 `pip check`。

## 9. 实施顺序与停止条件

实施必须按 T1 到 T5 逐节交付，每节单独 Review、提交和推送：

1. T1 先建立真实测量，未取得可信基线不得宣称优化；
2. T2 只处理 SDK/连接/分页和线程生命周期；
3. T3 只处理列式全市场与 dirty 定向提交；
4. T4 单独处理本地优先发布和 DeepSeek completion；
5. T5 最后处理浏览器逐行绘制。

每节必须保持业务哈希、固定融合、冻结和失败降级一致。某项优化若未达到绝对门限、引入
不受控线程/内存、改变冻结结果、扩大 DeepSeek 预算或只能依赖一次偶然计时，应回退该完整
章节并保留上一版本，不得用放宽实时性门限掩盖回退。

## 10. 剩余风险

- 供应商网络、限流和数据质量属于外部风险；本地优化只能减少排队、连接、解析和重复计算。
- Session 复用需要严格的线程归属和停止顺序，否则可能引入资源泄漏。
- dirty 更新必须保留全量重算等价性，不能用更快但不确定的哈希替代规范身份。
- `local_pending` 会让用户更早看到仅经本地风险保护的结果，页面必须明确状态，冻结前必须
  终态化。
- 固定离线 fixture 和浏览器基准不证明真实投资收益，也不替代真实交易日的来源年龄观测。
