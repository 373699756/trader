一、整体判断
代码整体质量高（冻结一致性、CAS、原子写入、分类 exception、有限的 worker/queue 都做得很好），但架构与性能上有若干结构性问题。最严重的是 RecommendationPipeline 已经变成"上帝类"和"上帝文件"，同时 MarketFeatureService 和 SnapshotRepository 也类似。模块边界在多个地方被打破，应用层通过私有属性 (pipeline._state, pipeline._publisher) 深度耦合了底层实现。

三、性能问题
3.1 SQLite 连接频繁 open/close
SnapshotRepository 中每个方法（publish, freeze, _stage_manifest, _commit_manifest, reserve_event, compare_and_set_event, list_events, pending_priority_events, record_data_source_health, latest, load_frozen, recommendation_dates, save_live_overlay, load_live_overlay）都 with connect(self._database_path) as connection: ...。

一次 _worker_loop 完整事件可能触发：

persist(reserve_event) → 开 1 个连接
persist(compare_and_set_event, "pending"→"running") → 开 1 个连接
处理逻辑中可能多次 persist → 各开 1 个连接
persist(publish, snapshot) → 开 1 个连接
persist(compare_and_set_event, "running"→"success") → 开 1 个连接
后台 _record_health_snapshot → 又开 1 个连接
粗略估算：每事件 5-10 个连接，每次 open/close ~0.5-2ms，热点期累计 5-20ms 浪费在 SQLite 句柄上。

修复建议：在 SnapshotRepository 内做"per-thread connection cache"或"long-lived writer thread"（已有 persistence_pool 但没利用），把 connect 限制在单线程。

3.2 SnapshotRepository.publish 每次都重新计算 sha256
python
应用
def publish(self, snapshot: RecommendationSnapshot) -> None:
    payload = snapshot_bytes(snapshot)   # 完整 JSON 序列化
    digest = snapshot_sha256(payload)     # 完整 sha256
    ...
snapshot_bytes 已经对 30+ recommendations 完整序列化，但同一快照可能在同一交易日内被发布多次（scoring 周期内更新），但每次都重新计算 hash。建议：

缓存 (snapshot.snapshot_id, payload, digest) 元组
写盘后立即复用
3.3 RecommendationReplayInput 包含冗余的"大对象"导致 frozen snapshot 巨大
replay_input.candidate_features 包含完整 FeatureSnapshot（每个 ~2KB），写盘时序列化到 frozen/.../*.json。一个 frozen snapshot 平均 30-100KB（行情是 5000+ 标的），JSON 解析每次 5-50ms。

_preselection_replay_feature 只挑了部分字段写 market_features，但 candidate_features 没做精简
replay 验证时会再 parse 一次完整 JSON + 完整 _restore_policy 重建 dataclass
修复建议：

RecommendationReplayInput 拆为 frozen_evidence（只存 hash）和 frozen_features（精简版）
反序列化时按需重新构建
3.4 MarketFeatureService 单一锁保护 4 个子缓存
python
应用
self._lock = threading.Lock()
_load_research 持锁时还要 pool.submit(...) + wait(futures)——会阻塞所有 4 个子系统的并发读。_load_intraday 同理。

修复建议：拆锁到 4 个独立 RLock，或用 threading.RLock 粒度化到 dict。

3.5 BoundedExecutor.submit 高频路径上的双重同步
python
应用
def submit(...):
    with self._lock:                        # 锁 1
        ...
        if not self._slots.acquire(blocking=False):  # 信号量
            return None
        self._submitted_count += 1
        self._inflight += 1
        ...
    try:
        future = executor.submit(run_tracked)  # ThreadPoolExecutor 内部还有锁
    except BaseException: ...
    future.add_done_callback(self._complete)  # 回调又加锁 1
    return future
每次 submit 都至少加 2 次 self._lock。在 6 worker / 数千次/分钟的 pipeline 中，这成为热点。

修复建议：用 threading.Lock + 单独 atomic counter（_inflight / _submitted_count 用 threading.local 或更细的"每 worker counter"），或直接用 concurrent.futures.ThreadPoolExecutor + 自定义 semaphore。

3.6 EastmoneyClient.fetch_market 同步等待所有分页
python
应用
remaining.extend((futures[future], future.result()) for future in as_completed(futures))
as_completed 顺序返回，第一个慢的 future 会阻塞全部——直到所有 page 都返回。fetch_market 整过程可达 5-10 秒（受最慢页 + 队列延迟影响）。

修复建议：改成 as_completed 配合"凑够 N 页就开始 normalize"（流式），或者预计算分页号并行发出去，gather 时按 page 顺序即可。

3.7 MarketDataGateway.fetch_market 用 _SingleFlight 但版本对比错
python
应用
if current is None or _quote_version(quote) > _quote_version(current):
    self._latest_by_code[quote.code] = quote
MarketQuote.quote_version 是 (source_time, received_time, data_version)——但每条行情都是新对象，source_time 经常相同（同一秒），导致大部分更新被丢弃成"已存在的版本"。建议用 id(quote) + 时间 + monotonic counter，或在 quote 上加 seq。

3.9 BoundedEventQueue._push_locked 偶发 O(n) 重建

if len(self._heap) > self._maximum_size * 2:

3.15 MarketFeatureService 中 LRU 淘汰走 O(n log n) sorted
python
应用
excess = len(self._candidate_quotes) - self._intraday_cache_limit
if excess > 0:
    for code in sorted(

每次 cache 满都排序全表。改用 collections.OrderedDict + move_to_end 维护 LRU 顺序。

3.16 pipeline.status() 持有 self._cadence_lock + self._merge_status_lock 等多锁
每个 status 调用都加 3+ 锁，与 worker hot path 抢锁。

3.17 SSE 慢消费者处理
event_stream_response 检查 is_subscribed，但心跳循环里没有 close() 检查，slow client 时 subscriber.get(timeout=...) 可能因 generator 关闭失败抛出 generator exit 异常。

3.18 SnapshotRepository._quarantine_orphans 启动时遍历所有 .json
for path in self._frozen_dir.rglob("*.json") 在大目录上可能数秒。建议：仅扫描 frozen/<strategy>/<date>/*.json 三层目录，跳过 orphan scan 或缓存结果。

3.19 调度 + worker 池共 6 个独立 ThreadPoolExecutor
bootstrap.py 创建 data_pool、persistence_pool，pipeline.py 再创建 normalization_pool、strategy_pool、deepseek_pool、long_pool、persistence_pool(注入)、merge worker thread。总计至少 7 个 ThreadPoolExecutor + 1 个独立 Thread。

ThreadPoolExecutor 内部各持 GIL 竞争、锁、idle timeout，跨过 GIL 切换。合并相近 pool 可减线程总数（建议保留 4 个：data/strategy/deepseek/persistence + merge 线程）。

3.20 SnapshotRepository._commit_manifest 一次性 INSERT N 行
python
应用
for recommendation in snapshot.recommendations:
    database.execute("INSERT INTO recommendations(...) VALUES (?, ?, ?, ?, ?, ?, ?)", ...)
TopK 是 10-18，但如果是 today + 多次 publish，每个 snapshot 都重插。可以 executemany 一次性插入并减少 fsync。

3.21 _atomic_replace 与 _atomic_create_immutable 每次 fsync 一次
发布期频繁（每 1-5s 一次），fsync 在 Windows / 机械盘上是 1-10ms。可以合并发布窗口（debounce）来减少 fsync 次数。

4.1 BoundedExecutor._worker_idents 在 stop 后未清理

4.2 MarketFeatureService._worker_pool.is_running() 与 submit TOCTOU

if pool is None or not pool.is_running() or pool.owns_current_thread():

future = pool.submit(function, *args, **kwargs)  # 此间 pool 可能被 stop

RuntimeState 自带锁，但 pipeline 又有 self._cadence_lock、self._merge_status_lock、self._lifecycle_lock，任何方法都会按不同顺序获取。死锁风险存在。

五、类型与接口问题
5.1 PipelineEvent.payload: Mapping[str, object] 下游不安全
event.payload.get("freeze_strategies") → 类型 object，多处 tuple[str(value) for value in freeze_raw if isinstance(freeze_raw, (list, tuple))——应该在事件层有 typed payload 子类。

5.2 RecommendationReplayInput.candidate_features: tuple[FeatureSnapshot, ...]
应限制为 PreselectionReplayFeature（已有但未使用）以减少反序列化负担。

5.3 Policy 用 MappingProxyType + __post_init__
每次 frozen=True dataclass 创建都要 __post_init__ 内 setattr，type checkers 不识别。

5.4 dataclass 字段顺序与默认值的不可序列化
_ResearchEntry、_HistoryEntry、_IntradayEntry 使用私有 dataclass 但字段都简单——可改为 NamedTuple 减少内存。

5.5 self._worker_idents: set[int] 在测试场景下泄漏
set 是无界的，热启动时 worker ident 会进入 set 但 stop 时未清空。

六、可观测性 / 错误处理
6.1 MarketFeatureService 几乎无 logging
只在 _load_research / _load_intraday 有隐式错误计数字段 _research_last_error。无 traceback 记录，调试时找不到根因。

6.2 多个 except BaseException 过宽
bootstrap.py 启动 pool 时 except BaseException，会捕获 KeyboardInterrupt / SystemExit，导致进程级中断信号也被吞掉（虽然后续会 re-raise，但已经执行了副作用）。

6.3 record_error 写入字符串但有截断
多处 pipeline._state.record_error(f"market data degraded during {phase.value}: {reason}") 截断到 500 字符，丢失上下文。建议同时记录 exc_type / exc_module。

6.4 无统一的 correlation_id
事件、DeepSeek 调用、SQLite 写、HTTP 请求没有共同的 trace_id，事故时无法在跨子系统间串起一次"业务请求"。
