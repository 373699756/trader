# 三板并行数据与差异化荐股完整实施计划

## 1. 文档定位

本文是对当前讨论结果的执行计划，不是新的业务契约。正式业务口径仍以
`docs/need.md` 为准；在对应实现批次完成、测试通过、Review、提交并推送前，
本文中的 v15/v16 机制均不得被运行配置或活动 API 直接启用。

本计划吸收 `TradingAgents-CN` 的多数据源覆盖和研究维度拆分，但不复制其自由文本信号解析、目标价推算、开放式多智能体辩论或无界 SDK 调用。

## 2. 已确定的范围与决策

- 当前生产基线为 v14；第一批建立 v15 数据与结构化合并边界，第二批建立 v16 三板评分与统一合并边界。
- 首期只改造 `tomorrow` 和 `d25`；`today`、`long`、冻结时间、DeepSeek 预算、HTTP 路径和固定融合公式保持不变。
- 东方财富、新浪、腾讯、Tushare、AKShare分别采集；Tushare为可选增强源，不承担高频实时报价。
- 沪深主板、创业板、科创板分别使用独立评分 worker 和独立策略参数，之后再由单一合并器统一排序和发布。
- 同行差和换手冲击采用趋势延续方向：正同行差、适度量价确认加分，极端换手不得继续抬分。
- d25 的质量价值组件按质量50%、价值30%、成长20%构成。
- 第二批动作门槛为 `tomorrow=78`、`d25=76`，观察带保持5分。
- DeepSeek继续采用受控主审+挑战者；模型只能解释经本地校验的结构化事实，不能新增股票、修改因子、直接扣分或决定动作。
- 本计划不做收益承诺、产品内回测、自动调参、机器学习训练、组合优化或真实下单；“更有收益”只能作为待验证目标。

## 3. 总体运行结构

```text
东方财富 ─┐
新浪     ─┼─> SourceObservation ─> CanonicalMarketSnapshot ─┐
腾讯     ─┤                                                  ├─> 全局复核规划器
Tushare  ─┤                                                  │       │
AKShare  ─┘                                                  │       v
                                                           ├─> DeepSeek结果回填
主板 worker ────────────────────────────────────────────────┤
创业板 worker ──────────────────────────────────────────────┤
科创板 worker ──────────────────────────────────────────────┘
                                                                  |
                                             单一合并器：融合/门槛/TopK/冻结
```

数据源和板块 worker 都是运行时生命周期内的有界执行单元；`create_app()` 仍然无线程、无网络、无数据库和无文件写入副作用。

## 4. 第一批：v15 数据源并行采集与结构化合并

### 4.1 独立数据源 worker

每个来源拥有独立 lane、single-flight、短期缓存、连续失败熔断、延迟统计和停止信号。队列容量固定为1，同源任务运行时只保留最新请求，不积压周期任务，不为单只股票创建线程。

来源职责固定如下：

| 来源 | 主要职责 | 频率边界 |
| --- | --- | --- |
| 东方财富 | 全市场实时行情、分钟、证券研究接口 | 按 `need.md` 高频周期刷新 |
| 新浪 | 全市场行情失败校验与回退 | 与全市场行情并行刷新 |
| 腾讯 | 候选池和 TopK 定向实时报价 | 仅按候选/TopK 周期刷新 |
| Tushare | 证券主数据、交易日历、前复权日线、日度估值、财务指标 | 盘前、收盘后或缓存到期时刷新 |
| AKShare | 行业、新闻、公告、财务和历史研究数据 | 按研究 TTL 和候选范围刷新 |

Tushare 以可选依赖 `tushare>=1.4,<2` 接入；未安装、无 Token、额度不足或
请求失败时，免费数据源链路继续工作并记录 `disabled` 或 `degraded`，不得阻塞
本地推荐。

### 4.2 单源观测与合并

每次来源返回先转换为 `SourceObservation`，至少包含：

- 代码、字段值和字段缺失原因；
- `source_time`、`received_time`、`effective_at`、时区和 `data_version`；
- 来源名称、请求批次、延迟、状态、错误类别和原始载荷摘要哈希；
- 该来源的完整度、数据年龄和是否允许参与合并。

纯合并器将同一观察点前的有效观测生成 `CanonicalMarketSnapshot`，至少保存
逐字段来源、参与版本、冲突、缺失、降级原因和 `merge_epoch`。合并不要求所有
来源同时返回；每个来源完成即更新候选观测，评分只读取满足新鲜度和版本约束
的完整 epoch。迟到或乱序结果只能进入审计，不能覆盖更新版本。

字段裁决固定为：

- 全市场价格和基础行情：同一时点优先东方财富，新浪用于独立校验和回退；
- 候选及 TopK 当前价格：在新鲜度合格时优先腾讯；
- 行业、财务、公告和研究字段：按数据类别选择有效来源，不以旧行情覆盖新行情；
- 多源价格偏差超过 `0.5%` 且未复核时只能观察；
- 所有来源失败时保留最近有效快照，并显式标记 `stale`/`degraded`，不得清空推荐。

### 4.3 板块身份和第一批风险门

板块识别顺序固定为：Tushare 生效证券主数据、AKShare 证券清单、行情市场字段、六位代码前缀降级。保存 `board`、`board_source`、`board_reliability`、交易所、上市日期、上市交易日龄、涨跌幅限制、规则版本和生效时间。

- 主板代码归入 `main`，创业板归入 `chinext`，科创板归入 `star`；不支持代码为 `unsupported`。
- 上市第1-5个交易日、重新上市首日和退市整理首日直接排除；第6个交易日起恢复普通规则。
- 上市日期缺失、主数据与行情字段冲突、代码前缀降级时只能观察并保留审计码。
- 主板 `8.00%` 通过、`8.01%` 剔除；创业板和科创板 `16.00%` 通过、`16.01%` 剔除。
- 无涨跌幅限制状态不得计算普通 `limit_proximity`。

第一批新增板内同行差、领先差、换手冲击、成交额冲击、流动性 P50/P80、样本数、回退交易日和可靠度输入，但不改变 v14 的分数、TopK 和动作门槛。

## 5. 第二批：三板独立评分与候选机制

### 5.1 三个独立评分 lane

每个板块各有一个 worker lane，消费同一不可变 `merge_epoch`，独立完成该板块
的候选预选、因子计算、风险审计和 tomorrow/d25 本地评分。lane 队列容量为1；
空候选是成功结果，失败不得静默转换为空候选。

所有板内因子只在 `trade_date + phase + board + data_version` 总体中截尾、归一化和排名。同行至少10只，领先组至少3只，且必须排除目标股票；板内样本不足100只时只允许回退最近有效交易日，回退超过5个交易日后该板只能观察。

### 5.2 候选预选权重

| 板块 | tomorrow | d25 |
| --- | --- | --- |
| 主板 | 流动35%、同行15%、趋势25%、稳定15%、完整10% | 流动30%、残差20%、趋势20%、稳定15%、执行10%、完整5% |
| 创业板 | 流动20%、同行30%、趋势25%、稳定15%、完整10% | 流动20%、残差30%、趋势20%、稳定10%、执行15%、完整5% |
| 科创板 | 流动25%、同行15%、趋势30%、稳定20%、完整10% | 流动25%、残差15%、趋势30%、稳定15%、执行10%、完整5% |

### 5.3 本地评分权重

| 板块 | tomorrow | d25 |
| --- | --- | --- |
| 主板 | 流动25%、动量10%、趋势20%、历史20%、执行15%、尾盘10% | 动量20%、趋势30%、流动25%、执行10%、不过热15% |
| 创业板 | 流动15%、动量25%、趋势20%、历史10%、执行20%、尾盘10% | 动量30%、趋势25%、流动20%、执行15%、不过热10% |
| 科创板 | 流动20%、动量10%、趋势25%、历史20%、执行10%、尾盘15% | 动量15%、趋势35%、流动25%、执行10%、不过热15% |

同行趋势方向为正同行差优先，领先差越小得分越高。换手冲击使用
`B(x;0.8,1.1,2.0,4.0)`，成交额冲击使用 `B(x;0.8,1.2,3.0,6.0)`。缺失
原始值保持 `null`，评分按中性50参与，同时降低 `board_data_reliability`；
可靠度低于 `0.85` 只能观察。

d25 删除 `base_score * overheat_factor * market_regime_factor` 双乘数。科创板质量价值组件固定为质量50%、价值30%、成长20%，其他板块使用已登记的结构化因子，不允许运行时自行换因子。

## 6. DeepSeek、融合与三板结果合并

### 6.1 全局复核协调

三个板块先完成本地评分，再由全局复核规划器按风险变化、动作门槛边界、TopK边界和证据冲突形成统一候选序列。DeepSeek 的预算、缓存、主审和挑战者状态全局共享，不能为每个板块复制预算或绕过188次物理请求上限。

主审和挑战者只能读取 `CanonicalMarketSnapshot`、板块身份、板内统计和证据
manifest；不能新增股票、重写本地数值、直接生成风险扣分、改变 veto 或决定
动作。

### 6.2 合并 epoch 和故障处理

每个板块输出 `BoardScoreBatch`，状态为 `success`、`empty`、`degraded` 或
`failed`。最终合并只接受交易日、阶段、策略版本、配置版本、因子 schema、
数据版本和 `merge_epoch` 全部一致的批次。

- 三个板块均成功或为空时，合并器生成完整候选集合。
- 单板失败或 epoch 不一致时，不发布偏置的新 TopK；继续保留最近完整快照并标记降级。
- 冻结只使用截止前满足年龄要求的最近完整三板结果；没有完整结果时不得用迟到数据拼造冻结记录。
- DeepSeek 失败、超时、弃权或预算耗尽时按现有 `local_degraded` 规则处理，不改变板块本地风险事实。

合并器随后统一执行：

1. 固定融合
   `clamp(local_score*0.68 + deepseek_score*0.32 - deepseek_risk_penalty, 0, 100)`；
2. `tomorrow >= 78`、`d25 >= 76` 的动作门槛和5分观察带；
3. 单板最多10只；主板同竞争组最多3只，创业板和科创板最多2只；
4. 默认 TopK 10、接口上限18；最终分、本地分、代码稳定排序；
5. 保存板块原排名、全局排名、被跳过候选、限制值和跳过原因。

## 7. 接口、配置和持久化

新增配置项：

- `board_candidate_weights`；
- `board_local_strategy_weights`；
- `board_factor_overrides`；
- `board_policy_version`；
- 各来源 lane 的 worker、TTL、超时、熔断和并发上限。

配置启动时必须校验三板与 tomorrow/d25 组合完整、每组权重精确合计1.0、因子已登记、版本参与策略哈希；禁止静默回退通用权重。

快照、SQLite、JSON 和只读 API 以加法增加：`board`、`board_policy_id`、板内
分位、板内排名、`board_data_reliability`、逐字段来源、`BoardScoreBatch` 状态、
`merge_epoch`、全局合并版本和选择跳过记录。旧 v14/v15 快照按原 schema 和
算法复算，不在读取时迁移。

## 9. 约束与剩余风险

本计划保证数据来源隔离、板内评分隔离和最终合并可复盘，但不保证任何板块
权重在所有市场环境中提高收益。交易成本、流动性、覆盖率、数据源波动、因子
相关性和 DeepSeek 可用性仍是外部风险；在另行建立收益验证契约前，不得把
分数、置信度或推荐数量解释为收益概率。
