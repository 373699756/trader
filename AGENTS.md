# AGENTS.md

本文件适用于整个仓库。所有自动化代理和人工协作者在修改本项目时都必须遵守。

## 1. 项目目标与运行范围

- 产品是仅运行在个人 PC 上的本地 A 股研究看板，不是多租户 SaaS，不提供真实下单能力。
- 目标浏览器为当前稳定版 Chrome、Edge 或 Firefox 桌面版；验收分辨率为 1280x720、1440x900 和 1920x1080。
- 手机和平板浏览器不属于产品范围，不得为了移动端增加新的业务分支或运行依赖。已有窄屏 CSS 可以保留，但不作为发布门禁。
- `docs/software-business-design.md` 是产品、架构、时间线、API、运维和验收契约的唯一权威；`docs/recommendation-strategy.md` 是候选、过滤、公式、风险、融合和排名策略的唯一权威；`pyproject.toml` 是依赖、构建和入口的唯一权威。

## 2. 固定架构边界

- 活动产品代码只能位于 `src/trader`。
- 依赖方向固定为 `entrypoints/web/infra -> application -> domain`。
- `domain` 不得读取配置、时钟、网络、文件或数据库。
- `application` 不得导入 `infra`、Flask 或旧包。
- `web` 只能调用应用层只读用例；HTTP 请求不得抓行情、评分或调用 DeepSeek。
- `bootstrap.py` 是唯一组合根。禁止全局服务定位器和隐藏构造外部客户端。
- `create_app()` 必须保持无线程、无网络、无数据库和无文件写入副作用。
- 新代码不得导入 `stock_analyzer`。v2 切换完成后，活动树中不得保留旧业务实现、旧测试或旧 Web 资源。

## 3. 不可破坏的业务约束

- 融合公式固定为 `clamp(local_score * 0.68 + deepseek_score * 0.32 - deepseek_risk_penalty, 0, 100)`，最终使用 `ROUND_HALF_UP` 保留两位小数。
- `local_score` 已扣本地风险，融合时禁止再次扣除 `local_risk_penalty`。
- DeepSeek 自由文本不得直接决定扣分；只允许本地规则映射经 schema 和证据校验的风险事实。
- DeepSeek 每日物理 HTTP 请求全局上限为 188，重试和失败同样计数；共享预热与 emergency 必须使用独立原子预算桶。
- today 在 11:20 冻结，tomorrow/d25 在 14:50 冻结。冻结后任何迟到结果都不得修改记录或 JSON。
- long 只展示当前快照，不冻结、不写推荐历史。
- 所有业务时间使用 `Asia/Shanghai`，时钟必须可注入且带时区。
- 数据源或 DeepSeek 失败不能阻塞本地推荐和只读 Web；必须保留最近有效快照并显式标记降级。

## 4. 强制执行流程

每个用户任务都是一个独立交付批次，必须形成并持续更新可验证计划。一次只允许一个计划项处于 `in_progress`，不得把不同目标混入同一批次。

### 4.1 “继续”命令与整节任务边界

1. 用户每发送一次“继续”或语义等价的继续执行指令，都视为一个新的独立交付批次；该批次必须处理计划或需求文档中下一个完整未完成章节，以同级标题为章节边界，并完成章节内全部明确子项。
2. 同一章节可以包含多个可独立验收子项，必须作为一个整节任务统一实现、回归、Review、提交和推送；不得完成首个子项后提前停止，也不得顺带处理相邻章节或无关优化。
3. 开始前记录当前 `HEAD`、上游分支、工作树已有变更和本任务文件范围。此前批次如尚未完成 Review、提交或推送，必须先闭合此前批次，不能把两个批次合并。
4. 当前整节任务完成全部子项、Review、提交并成功推送后必须停止，不得自动执行计划中的下一章节；等待用户下一次“继续”。
5. 用户直接下达的新任务仍形成独立交付批次；用户明确指定完整章节时按整节交付，否则多个互不相关目标仍拆成多个任务逐次交付。

### 4.2 实施

1. 先阅读 `AGENTS.md`、`docs/software-business-design.md`、`docs/recommendation-strategy.md`、相关代码、测试、配置和当前 `git status`。
2. 只修改当前计划项需要的边界，保留用户已有和无关变更。
3. 先修改契约与测试，再修改实现；修复缺陷时必须补回归测试。
4. 所有手工文件修改使用 `apply_patch`。禁止使用破坏性 Git 命令。
5. 不提交密钥、运行数据库、快照、缓存、构建产物或浏览器临时文件。

### 4.3 每个独立任务完成后的 Review 循环

独立任务实现完成后，不能直接标记完成。必须以上一次已推送提交为基线，针对本任务完整 diff 执行以下循环：

1. 查看该计划项的完整 diff 和新增文件，确认没有越界改动、死代码、重复实现或遗留 TODO。
2. 按正确性、冻结一致性、并发/资源生命周期、错误降级、安全、类型、API 兼容、桌面 UI 和可安装性逐项审查。
3. 运行与风险相称的最小测试、Ruff 和 mypy；共享边界变更必须运行全部新包测试。
4. 记录所有发现，立即修改代码与测试。
5. 重新查看 diff 并重复审查与验证，直到没有已知审查发现且所有适用门禁通过。
6. 只有满足上述退出条件后，才把 Review 计划项标记为 `completed`；独立交付任务在提交成功推送并核对上游前必须保持 `in_progress`，整体计划 Review 不能替代本任务的独立 Review 证据。

“完美”在本项目中表示：没有已知未解决问题、需求逐项可追踪、所有适用自动门禁通过、剩余外部风险被明确记录；禁止用主观判断代替证据。

### 4.4 每个独立任务完成后的提交与推送

1. 对当前任务 diff 再做一次独立最终 Review，并修复到零已知发现。
2. 更新根目录 `CHANGELOG.md` 的 `Unreleased`，详细写明 Added、Changed、Fixed、Removed、Verification 和 Residual Risks。
3. 运行完整质量、测试、构建和仓库外 wheel 安装验收。
4. 用 `git status` 和 `git diff --check` 确认没有生成物、空白错误或遗漏文件。
5. 仅暂存本任务文件，使用 Conventional Commit 格式创建且只创建一个新提交；提交信息必须准确描述行为变化，禁止 amend、squash 或复用此前任务提交。
6. 提交后立即推送当前工作分支到其跟踪远端；首次推送使用同名分支建立上游。必须分别读取本地 `HEAD` 与 `@{upstream}` 并确认提交哈希一致，未成功推送的任务不得宣告完成，也不得开始下一任务。
7. 推送只包含当前任务提交。除非用户明确要求，不创建 PR、合并分支、打 tag 或发布版本。

### 4.5 用户问题与修改说明归档

1. 每个独立交付批次必须在根目录 `CHANGELOG.md` 的 `Unreleased` 中归纳用户提出的问题及对应修改说明，不能只罗列文件名、提交信息或测试命令。
2. 归纳至少覆盖用户可观察到的症状或明确诉求、原因或现状判断、实际修改及行为变化、验证证据、剩余风险与后续项；原因尚未确认时必须写明“待验证”，不得把推测写成事实。
3. 用户连续提出的多条反馈如属于同一根因可以合并归纳；互不相关的问题仍按独立交付批次分别记录，不得用笼统的“已优化”代替逐项对应关系。
4. 产品、架构、时间线、API、运维或验收契约变化必须更新 `docs/software-business-design.md`；候选、过滤、公式、风险、融合或排名变化必须更新 `docs/recommendation-strategy.md`；跨边界变化同时更新两份。归纳文档是交付记录，不得与权威契约形成第二套定义。
5. 文档不得写入密钥、令牌、完整外部请求/响应、个人路径中的敏感信息或运行数据；必要证据只保留脱敏摘要、错误类别、计数、版本和验证结论。

## 5. 质量门禁

最终目录切换后统一使用：

```bash
make format-check
make lint
make type-check
make test
make package
```

还必须验证：

- 架构 AST 契约和 `create_app()` 无副作用契约。
- 固定融合向量结果为 `83.40`。
- 预算并发上限、SSE 游标恢复、慢客户端、冻结恢复和哈希一致性。
- 从仓库外安装 wheel 后可导入包、执行 `trader-cli`，并读取模板、CSS、JavaScript 和图标资源。
- 桌面 1280x720、1440x900、1920x1080 无白屏、重叠、页面级横向溢出或明显布局跳动。

## 6. 编码与文档规范

- 支持 Python 3.10-3.14，公共边界和复杂返回值必须有真实类型标注。
- 沿用 Ruff、mypy、pytest 和 `src` 布局；不要引入第二套格式化、依赖或配置真相源。
- 核心计算保持纯函数；网络、SQLite、线程、文件和时钟通过边界显式注入。
- 外部 I/O 必须有超时、上限和明确失败策略；后台线程必须可停止、可等待、可观察。
- 捕获最窄异常，保留异常因果链，不记录密钥、完整外部载荷或敏感数据。
- 注释只解释非显然约束和原因；禁止留下无完成条件的 TODO。
- 业务契约变化必须同步更新对应的两份权威文档、测试和 `CHANGELOG.md`。

## 7. Git 与工作树安全

- 工作树可能包含用户未提交修改，默认全部保留并与之协作。
- 禁止 `git reset --hard`、`git checkout --` 或擅自删除用户变更。
- 提交前检查暂存区，避免把 `.runtime/`、`.venv/`、`dist/`、`build/`、缓存或截图带入提交。
- 每个独立任务保持一个可审计提交和一次对应推送；不得把多个“继续”批次压入同一提交，也不得用后续任务改写已推送历史。
- 推送失败时保留本地提交，记录失败原因并停止；恢复推送并核对上游后才能继续下一任务。
- 回退必须切换完整旧 release；v2 数据不得写回旧运行库。
